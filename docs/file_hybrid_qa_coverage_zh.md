# 文件 QA 与混合 QA 覆盖范围说明

## 1. 文档目的

这份文档只回答一个问题：

当前计划中的“文件 QA”和“混合 QA”，到底覆盖哪些场景，不覆盖哪些场景。

这份文档不讨论实现细节，只讨论范围与判定。

适用对象：
- 你用来检查需求是否被完整覆盖
- 后续 gateway 协议设计和 fastQA 执行设计的验收基线

---

## 2. 总体范围

本次只覆盖“文件参与的问答”。

包括三类路由：
- `pdf_qa`
- `tabular_qa`
- `hybrid_qa`

不包括：
- 普通 `kb_qa`
- 不带任何文件上下文的普通知识库问答
- highThinking 独立链路
- patent 链路

一句话概括：
- 只要这次回答要用到文件，就进入这套范围
- 如果完全不需要文件，就不在这次范围里

---

## 3. 三类路由的定义

## 3.1 `pdf_qa`

定义：
- 只基于 PDF 文件回答
- 不引入知识库补充
- 不引入表格执行结果

典型特征：
- 用户的问题只要求解释、总结、抽取、对比 PDF 文本内容
- 问题不要求结合知识库背景
- 问题不要求结合表格数据

## 3.2 `tabular_qa`

定义：
- 只基于表格文件回答
- 不引入知识库补充
- 不引入 PDF 文本证据

典型特征：
- 用户明确在问表格中的数值、排序、筛选、统计、聚合、趋势、对比
- 问题可以靠表格执行结果独立完成

## 3.3 `hybrid_qa`

定义：
- 只要有两类及以上信息源联合参与回答，就统一归入 `hybrid_qa`

本次 `hybrid_qa` 明确覆盖四种子场景：
- `pdf+kb`
- `table+kb`
- `pdf+table`
- `pdf+table+kb`

也就是说：
- 文件 + 知识库，属于 `hybrid_qa`
- PDF + 表格，属于 `hybrid_qa`
- PDF + 表格 + 知识库，也属于 `hybrid_qa`

---

## 4. 明确覆盖的 source scope

为了避免“route 看起来一样，但内部逻辑其实不同”，这里把实际覆盖的来源组合也列清楚。

### 4.1 单一来源
- `pdf`
- `table`

### 4.2 多来源
- `pdf+kb`
- `table+kb`
- `pdf+table`
- `pdf+table+kb`

当前不再推荐继续使用这种表达：
- `pdf_qa + allow_kb_verification`
- `tabular_qa + allow_kb_verification`

因为这种表达会让“单源路由”和“混合路由”语义混在一起。

---

## 5. 覆盖范围的核心判定原则

## 5.1 是否属于文件 QA 范围

一个问题进入本次范围，必须满足至少一条：
- 用户明确引用了某个文件
- 用户当前问题需要结合对话中的上传文件回答
- 用户的答案必须依赖某个 PDF 或某个表格
- 用户明确要求“结合论文/文件/表格”回答

如果以上都不满足，就不属于这次文件 QA 范围。

## 5.2 是否属于混合 QA

只要这次回答要联合两类或以上来源，就属于混合 QA。

这里的“来源”指的是：
- PDF
- 表格
- 知识库

因此以下都属于混合 QA：
- PDF + 知识库
- 表格 + 知识库
- PDF + 表格
- PDF + 表格 + 知识库

## 5.3 是否需要知识库参与

只有在用户明确需要以下能力时，才应进入带 `kb` 的混合问答：
- 解释文件中没有直接写清楚的背景机制
- 用知识库补充行业/领域背景
- 对文件中的结论做知识库层面的交叉验证
- 对文件结果做机理解释、原理解释、背景补充

不能因为问题“比较难”，就默认自动加知识库。

## 6. 明确覆盖的场景案例

下面按“应该覆盖”的场景列出示例。

## 6.1 纯 PDF 问答

### 场景 1：总结单篇论文
用户：
- “总结这篇论文的核心结论”

判定：
- `pdf_qa`
- 来源：`pdf`

原因：
- 只需要 PDF
- 不需要知识库
- 不需要表格

### 场景 2：抽取论文中的实验条件
用户：
- “这篇论文里的测试温度是多少”

判定：
- `pdf_qa`
- 来源：`pdf`

### 场景 3：比较两篇论文的表述
用户：
- “比较这两篇论文对于循环寿命的结论差异”

判定：
- `pdf_qa`
- 来源：`pdf`

说明：
- 虽然用了多个 PDF，但本质上仍然只用了 PDF 这一类来源
- 多 PDF 不等于混合 QA

---

## 6.2 纯表格问答

### 场景 4：找最大值/最优组
用户：
- “这个表里哪组效率最高”

判定：
- `tabular_qa`
- 来源：`table`

### 场景 5：统计/筛选
用户：
- “把容量大于 200 的样本筛出来，并按效率降序排列”

判定：
- `tabular_qa`
- 来源：`table`

### 场景 6：跨表对比，但只有表格
用户：
- “比较这两个表里同一种材料的倍率性能差异”

判定：
- `tabular_qa`
- 来源：`table`

说明：
- 即使有多个表格，只要没有 PDF 和 KB 联合参与，仍然属于纯表格问答

---

## 6.3 PDF + 知识库

### 场景 7：论文内容 + 背景解释
用户：
- “这篇论文里提到的电压窗口为什么这样设计，结合知识库解释一下”

判定：
- `hybrid_qa`
- 来源：`pdf+kb`

### 场景 8：论文结论 + 领域知识补充
用户：
- “论文里说这个掺杂能提升稳定性，结合知识库讲一下可能机制”

判定：
- `hybrid_qa`
- 来源：`pdf+kb`

### 场景 9：论文结果 + 机理交叉验证
用户：
- “结合这篇论文和知识库，看一下这个结论是否合理”

判定：
- `hybrid_qa`
- 来源：`pdf+kb`

---

## 6.4 表格 + 知识库

### 场景 10：表格结果 + 原因解释
用户：
- “这个表里循环寿命最差的是哪组，结合知识库解释原因”

判定：
- `hybrid_qa`
- 来源：`table+kb`

### 场景 11：表格数值 + 行业背景
用户：
- “这个表里高电压组性能下降更明显，结合知识库解释一下这个现象”

判定：
- `hybrid_qa`
- 来源：`table+kb`

### 场景 12：表格统计结论 + 机理说明
用户：
- “把这个表的趋势总结一下，并结合知识库说明背后的可能机理”

判定：
- `hybrid_qa`
- 来源：`table+kb`

---

## 6.5 PDF + 表格

### 场景 13：论文说法 vs 实验表格
用户：
- “结合这篇论文和这个表格，判断文中的结论和实验数据是否一致”

判定：
- `hybrid_qa`
- 来源：`pdf+table`

### 场景 14：论文中的指标解释 + 表格验证
用户：
- “论文里说倍率性能提高了，你结合这个表格看一下是不是这样”

判定：
- `hybrid_qa`
- 来源：`pdf+table`

### 场景 15：用论文方法解释表格结果
用户：
- “结合论文中的实验设计和这个表格结果，判断是哪一步影响最大”

判定：
- `hybrid_qa`
- 来源：`pdf+table`

说明：
- 这里没有用知识库
- 但已经有两类文件来源联合参与，所以仍属于混合 QA

---

## 6.6 PDF + 表格 + 知识库

### 场景 16：论文 + 表格 + 机理解释
用户：
- “结合论文、表格和知识库，判断这组设计是否合理，并说明原因”

判定：
- `hybrid_qa`
- 来源：`pdf+table+kb`

### 场景 17：文件事实 + 知识补充的完整联合判断
用户：
- “结合这篇论文、这个实验表和知识库，分析这组材料为什么性能最好”

判定：
- `hybrid_qa`
- 来源：`pdf+table+kb`

### 场景 18：文件交叉验证 + 背景解释
用户：
- “结合论文里的说法、表格里的数据和知识库背景，判断这个结论是否站得住”

判定：
- `hybrid_qa`
- 来源：`pdf+table+kb`

---

## 7. 明确不覆盖或不属于本次范围的场景

## 7.1 纯普通问答

### 场景 19：不涉及任何文件
用户：
- “锂电池高电压窗口为什么容易带来副反应”

判定：
- 不属于本次文件 QA / 混合 QA 范围
- 属于普通 `kb_qa`

原因：
- 没有文件参与

## 7.2 仅仅提到“论文/文件”但并未真正依赖文件

### 场景 20：泛泛而谈
用户：
- “论文一般怎么写实验结论”

判定：
- 不属于本次文件 QA / 混合 QA 范围

原因：
- 虽然提到了“论文”，但不是在问当前会话中的具体文件

## 7.3 用户只是在文件会话里，但当前问题不需要文件

### 场景 21：会话里有文件，但问题与文件无关
用户：
- “顺便问一下，什么是 CEI 膜”

判定：
- 不属于文件 QA
- 应回到普通 `kb_qa`

原因：
- 会话里有文件，不等于每个问题都必须走文件 QA

## 7.4 单纯多文件但仍是单一来源

### 场景 22：多篇 PDF 对比
用户：
- “比较这三篇论文的结论”

判定：
- `pdf_qa`
- 不是 `hybrid_qa`

原因：
- 仍然只有 PDF 一种来源

### 场景 23：多张表格联合统计
用户：
- “把这两个表合起来算平均值”

判定：
- `tabular_qa`
- 不是 `hybrid_qa`

原因：
- 仍然只有 table 一种来源

---

## 8. 容易混淆但必须明确覆盖的边界情况

## 8.1 “PDF + 普通问题”其实是混合 QA

例子：
- “我在这个论文的总结里看到电压问题，你结合论文和知识库回答一下”

结论：
- 这不是 `pdf_qa`
- 这是 `hybrid_qa/pdf+kb`

## 8.2 “表格 + 普通问题”其实也是混合 QA

例子：
- “这个表里高电压那组差很多，结合知识库解释原因”

结论：
- 这不是 `tabular_qa`
- 这是 `hybrid_qa/table+kb`

## 8.3 “PDF + 表格”就算没有 KB，也已经是混合 QA

例子：
- “结合论文和表格看一下结论是否一致”

结论：
- 这是 `hybrid_qa/pdf+table`
- 不是 `pdf_qa`
- 也不是 `tabular_qa`

## 8.4 “多 PDF”不是混合 QA

例子：
- “比较这两篇论文的实验设计差异”

结论：
- 仍然是 `pdf_qa`

## 8.5 “多表格”不是混合 QA

例子：
- “比较这两个 excel 的结果差异”

结论：
- 仍然是 `tabular_qa`

---

## 9. 文件选择相关场景覆盖

这部分不是在定义执行路由，而是在定义“哪些文件选择场景也在本次体系里必须覆盖”。

明确需要覆盖：
- 明确编号引用
  - 例如 `#1`、`#2`
- 序号引用
  - 例如“第一个文件”“第二篇论文”
- 最近上传引用
  - 例如“刚上传的那个”“最新上传的论文”
- 单数模糊引用
  - 例如“这篇论文”“这个表格”
- 复数范围引用
  - 例如“这些论文”“所有表格”
- 上轮 focus 复用
  - 例如上一轮已经在问某篇论文，这一轮继续问“这篇论文里还有什么”
- 删除文件后的澄清
- 多候选文件时的澄清

说明：
- 这些属于 gateway 的覆盖范围
- 它们不是 fastQA 的二次判断范围

---

## 10. 典型产品视角检查表

如果你要快速判断是否覆盖到位，可以用下面这个检查表。

### 10.1 PDF 问答是否覆盖到位
- 单篇 PDF 总结
- 单篇 PDF 信息抽取
- 多篇 PDF 对比
- 单篇 PDF 引文/结论定位

### 10.2 表格问答是否覆盖到位
- 单表统计
- 单表筛选/排序
- 多表比较
- 表格结果总结

### 10.3 混合问答是否覆盖到位
- PDF + KB
- table + KB
- PDF + table
- PDF + table + KB

### 10.4 边界是否覆盖到位
- 模糊文件引用
- 删除文件引用
- 最新上传文件引用
- 上轮 focus 继承
- 文件会话中切回普通问答

---

## 11. 建议的最终理解

你可以把这次范围理解成：

### 第一层：是否需要文件
- 不需要文件 -> 普通 `kb_qa`，不在本次范围里
- 需要文件 -> 进入本次范围

### 第二层：用了几类来源
- 只用 PDF -> `pdf_qa`
- 只用 table -> `tabular_qa`
- 用了两类及以上来源 -> `hybrid_qa`

### 第三层：混合 QA 具体是哪种组合
- `pdf+kb`
- `table+kb`
- `pdf+table`
- `pdf+table+kb`

这是当前最清晰、最不容易出边界歧义的设计。

---

## 12. 当前文档结论

如果按这份覆盖文档执行，那么本次文件 QA / 混合 QA 的范围已经明确包含：
- PDF 单源问答
- 表格单源问答
- 文件 + 知识库问答
- PDF + 表格联合问答
- PDF + 表格 + 知识库联合问答
- 文件选择与澄清相关场景

并且明确排除：
- 纯普通 `kb_qa`
- 不依赖当前文件的泛泛而谈问题
- 仅因为当前会话里有文件就强制走文件问答的情况

如果你认这份文档的覆盖范围，就说明大的场景集合已经定住了，后面就可以按这个范围推进协议和任务执行。
