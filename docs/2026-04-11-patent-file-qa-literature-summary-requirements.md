# Patent 文件问答文献总结优化需求

**日期：** 2026-04-11

## 1. 背景

当前 `patent` 模式下的文件问答链路，在涉及 PDF 文献总结或多文件文献归纳时，答案结构仍明显偏向扁平化输出，常见表现为：

1. 以 `结论 / 证据 / 对比 / 限制` 四块固定模板回答
2. 内容层级浅，缺少学术化章节组织
3. 对研究背景、方法、结果、意义的展开不充分
4. 信息点密度不足，具体实验方法、结果数据和结论价值表达不稳定

对照 `fastQA` 当前 PDF 总结效果，`patent` 文件问答链路需要把“文献总结类回答”升级到更清晰、更完整、更适合阅读和比对的结构化输出。

---

## 2. 目标

本需求的目标是：

1. 将 `patent` 文件问答链路中的文献总结输出，从当前四段扁平模板升级为学术结构化总结
2. 使单篇 PDF 总结、多篇 PDF 比较中的逐篇概要、以及混合文件问答里的文献总结段落，都具备统一的高质量结构
3. 保持“严格基于文件原文证据、不允许引入文件外推测”的边界不变

---

## 3. 明确范围

### 3.1 本需求覆盖

本需求仅覆盖 `patent` 模式下的文件问答链路，包括：

1. `pdf_qa`
2. `hybrid_qa`
3. `tabular_qa` 中命中当前 summary/file-summary 判定的总结类问题输出部分
4. 多 PDF 比较场景中的逐篇文献概要与最终总结

### 3.2 本需求不覆盖

本需求不覆盖以下内容：

1. `patent` 普通问答主链路
2. `patent` 普通知识库问答的检索与生成逻辑
3. `fastQA` 普通问答或 `highThinkingQA` 普通问答
4. 前端 mode 入口、按钮和页面布局大改
5. 网关路由判定逻辑
6. `patent` 普通 QA 主链路文件，例如 `answering.py`、`stages/synthesis.py`

**约束：本需求实施时，不应改动普通问答代码。**

---

## 4. 当前问题

### 4.1 结构过于简单

当前 `patent` 文件问答里的总结型回答，在单篇 PDF、hybrid 和 summary 型 table 路径里，常见结构仅为：

1. 结论
2. 证据
3. 对比
4. 限制

该结构适合轻量提示，不适合承载“文献总结”这种需要背景、方法、结果、意义完整表达的回答任务。

补充说明：

1. 多 PDF compare 的 fallback 路径当前并不完全是这四段结构，而是另一套“逐篇概要 + 相同点/差异点/总结”的组织方式
2. 本需求的重点不是把所有文件问答都统一成一种死板模板，而是把“文献总结类输出”升级为更稳定的学术结构化表达

### 4.2 内容层次不清

当前回答通常是平铺式条目，缺少：

1. 学术逻辑上的章节划分
2. 一级与二级信息层级
3. 每个章节下的细化说明

### 4.3 信息密度不足

当前回答容易出现：

1. 只说“做了什么”，不说“为什么做”
2. 只说“有结果”，不说“具体结果是什么”
3. 只说“有结论”，不说“结论对领域意味着什么”

### 4.4 与 `fastQA` 的总结体验不一致

`fastQA` 的 PDF 总结更接近“文献研究内容总结”，而 `patent` 文件问答当前更像“证据式摘要壳子”，两者体验存在明显落差。

---

## 5. 目标输出形态

`patent` 文件问答中的文献总结类回答，应升级为类似以下结构：

1. 研究目的和背景
2. 研究方法/实验设计
3. 主要发现和结果
4. 结论和意义
5. 注释说明

### 5.1 章节要求

每次文献总结类回答，至少包含以下四个独立章节：

1. `研究目的和背景`
2. `研究方法/实验设计`
3. `主要发现和结果`
4. `结论和意义`

底部应追加统一说明：

`注*：所有总结内容均严格基于文件原文中明确提到的信息，未添加任何通用知识或推测内容。`

### 5.2 列表要求

输出列表要求如下：

1. 使用 bullet points，而不是 `1. 2. 3.` 编号主列表
2. 为保证当前前端 Markdown 渲染兼容性，应使用标准 Markdown 列表标记，如 `-`、`*` 或 `+`，不要求后端直接输出原始 `•` 字符
3. 支持二级嵌套
4. 证据充分时，每个章节应尽量展开为 3 个左右要点；若原文信息不足，允许少于 3 个，但不得补写或推测
5. 要点应尽量完整，不允许只有短句标签

建议形态示例：

```markdown
## 研究方法/实验设计

- 研究对象：
  - xxx 材料 / xxx 电极 / xxx 工艺对象
- 关键方法：
  - XRD
  - TOF-SIMS
  - S-cells
- 技术路线：
  - 先……
  - 再……
```

### 5.3 内容深度要求

#### 研究目的和背景

证据充分时，优先覆盖：

1. 本研究关注的核心问题
2. 当前已有方案或领域痛点
3. 本文要解决的具体矛盾、瓶颈或假设

#### 研究方法/实验设计

证据充分时，优先覆盖：

1. 研究对象
2. 方法、实验手段或分析工具
3. 技术路线或验证方式

#### 主要发现和结果

证据充分时，优先覆盖：

1. 关键实验现象
2. 定量结果、关键数值或对比结果
3. 与研究问题直接相关的核心发现

#### 结论和意义

证据充分时，优先覆盖：

1. 核心贡献
2. 对应用、工艺或领域理解的价值
3. 若原文提及，补充其局限或后续方向

---

## 6. 格式要求

### 6.1 输出格式

回答应优先使用 Markdown 结构化输出。

最低要求：

1. 有明确章节标题
2. 有符合 Markdown 规范的项目列表
3. 支持嵌套层级
4. 专业术语保留原文写法

### 6.2 样式目标

前端最终展示应朝以下方向对齐：

1. 章节标题有明显视觉分层
2. 标题区分于正文
3. 列表缩进清晰
4. 阅读时能快速分辨“背景 / 方法 / 结果 / 结论”

说明：

1. 标题蓝色、下划线等样式属于前端展示目标
2. 本需求首先要求后端输出具备可渲染的结构
3. 若前端已支持 Markdown 标题和列表，则不要求本轮改动页面主结构

---

## 7. 适用场景要求

### 7.1 单篇 PDF 总结

当用户提问类似以下问题时：

1. “总结这篇文献的主要内容”
2. “这篇文章主要讲了什么”
3. “概括一下这篇文献”

回答应输出完整的四章节总结，而不是仅输出 `结论 / 证据 / 对比 / 限制` 模板。

如果原文可提取信息不足，则仍应保持四章节框架，但允许个别章节只保留 1 到 2 个高置信要点，并明确收敛表达，不能为凑结构而补写。

### 7.2 多篇 PDF 比较中的逐篇概要

当用户提问类似：

1. “对比这两篇文献”
2. “分别总结这两篇文献的内容并比较”

则：

1. 每篇文献概要应尽量使用本需求规定的结构化总结逻辑
2. 不要求每篇都展开到极长，但至少应覆盖背景、方法、结果、结论中的关键信息
3. 比较部分应建立在逐篇概要基础上，而不是直接跳到笼统结论

### 7.3 混合文件问答

在 `pdf + table` 或 `pdf + table + kb` 场景中：

1. 文献总结部分仍应保持该结构
2. 在 `pdf + table` 场景里，PDF 原文证据与表格证据都可以作为主结论依据，不应把 table 降级为纯陪衬信息
3. 在 `pdf + table + kb` 场景里，知识库内容只作为补充、验证或交叉对照，不能伪装成 PDF 原文总结
4. 若 `tabular_qa` 的问题本质是字段抽取、数值查询或表格证据定位，而非 summary/file-summary 判定下的总结类问题，则不适用本需求的学术综述化输出

---

## 8. 非目标

以下内容不属于本需求目标：

1. 把所有 `patent` 文件问答回答都改成学术综述风格
2. 改动普通问答主链路 prompt
3. 放松“只基于原文”的证据边界
4. 用更长篇幅替代更好结构

本需求关注的是：

1. 文献总结类回答的结构化升级
2. 在不破坏证据边界的前提下提升表达层次

---

## 9. 验收标准

满足以下条件，才算达到需求：

1. 文献总结类回答包含至少 4 个独立章节
2. 章节至少包括：研究目的和背景、研究方法/实验设计、主要发现和结果、结论和意义
3. 输出支持二级嵌套列表，并使用当前前端可稳定渲染的标准 Markdown 列表标记
4. 证据充分时，各章节应尽量提供 3 个左右详细要点；证据不足时允许减少要点数量，但必须保守收敛，不得补写
5. 在原文确有信息时，回答中应出现具体方法、实验对象、现象或数值信息，而不是只有泛化概括
6. 底部带有“严格基于原文、不做推测”的注释声明
7. 实施范围仅限 `patent` 文件问答链路，不影响普通问答主链路
8. 不应把普通表格字段问答、数值查询问答强行改造成文献综述风格

---

## 10. 实施备注

从当前代码现状看，本需求应优先落在以下层面：

1. `patent` 文件问答 prompt 结构
2. `patent` 文件问答 fallback/包装逻辑
3. `patent` 混合问答中的文献总结输出模板
4. `tabular_qa` 中仅在当前 summary/file-summary 判定命中时的总结输出包装

不应把本需求解释为：

1. 改 `patent` 普通 QA 的阶段四答案生成
2. 改 `patent` 全模式回答格式
3. 改全站所有总结功能
4. 改普通表格字段问答、数值检索问答的输出形态
