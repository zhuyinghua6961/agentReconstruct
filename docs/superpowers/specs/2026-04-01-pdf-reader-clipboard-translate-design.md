# PDF Reader 一键粘贴并翻译 Spec

日期：2026-04-01

## 1. 背景

当前“查看原文”右侧面板中的翻译能力已经可用，但交互链路仍然偏手动：

1. 用户在 PDF 原文中复制文本
2. 用户切换到“翻译”子面板
3. 用户把文本手动粘贴到输入框
4. 用户点击“翻译文本”

这条链路能用，但操作偏繁琐。目标是在不破坏现有手动输入能力的前提下，增加一个更顺手的入口，让用户复制完文本后可以在翻译面板中一键完成“粘贴 + 翻译”。

## 2. 目标

新增一个前端交互入口：

- 保留当前手动输入框与“翻译文本”按钮
- 新增“粘贴并翻译”按钮
- 点击后读取系统剪贴板文本
- 读取成功时：
  - 把文本同步写入现有翻译输入框
  - 立即复用现有翻译请求链路发起翻译
- 读取失败或剪贴板无有效文本时：
  - 在翻译面板内给出友好、明确、非打断式反馈
- 新入口沿用现有翻译调用与配额语义：
  - 仍然走现有 `api.translate` 接口
  - 仍然按现有翻译能力的配额/限流结果处理
  - 不额外增加确认弹窗、二次扣费规则或新的 quota 分支

## 3. 非目标

本次不做以下事情：

- 不改后端翻译接口
- 不改 quota 计费/扣费逻辑
- 不做“自动监听系统复制事件”
- 不做跨页面全局剪贴板托管
- 不替换现有手动粘贴方式
- 不做 OCR、截图翻译、PDF 选区直连翻译

## 4. 现状实现

当前实现位于：

- [PdfReader.vue](/home/cqy/worktrees/highThinking/frontend-vue/src/components/PdfReader.vue)
- 翻译调用入口：
  - `manualText`
  - `translateSelected()`
- 翻译 API：
  - [api.js](/home/cqy/worktrees/highThinking/frontend-vue/src/services/api.js)
  - `api.translate(texts)`

当前翻译区的核心状态包括：

- `manualText`
- `translations`
- `isTranslating`
- `translationQuotaCard`

因此本需求本质上是一个 `PdfReader` 局部交互增强，前后端契约无需新增字段。

当前翻译请求的实际调用形态是：

- `api.translate([manualText.value])`

因此本次“粘贴并翻译”成功路径也应保持同一请求契约：

- 向翻译接口传入“只包含 1 个字符串元素的数组”
- 不按段落拆分
- 不按换行拆分
- 不新增批量翻译协议

## 5. 方案对比

### 方案 A：新增“粘贴剪贴板”按钮，用户再点“翻译文本”

优点：

- 行为清晰
- 风险最低
- 失败和回退最简单

缺点：

- 仍然是两步操作
- 改善幅度有限

### 方案 B：新增“粘贴并翻译”按钮，同时保留现有手动方式

优点：

- 最贴合真实使用路径
- 用户复制后只需一次点击
- 不影响现有手动输入兜底能力

缺点：

- 需要处理剪贴板权限失败、空文本、环境不支持等状态

### 方案 C：自动检测剪贴板变化并提示翻译

优点：

- 理论上更智能

缺点：

- 复杂度高
- 权限/兼容性问题多
- 容易引入过度打扰
- 超出当前需求

## 6. 结论

采用方案 B：

- 新增“粘贴并翻译”
- 保留现有“翻译文本”
- 两条路径共享同一个翻译执行逻辑
- 剪贴板读取失败时给出局部友好反馈

这是最符合用户体验目标、同时改动面最小的方案。

## 7. 交互设计

### 7.1 翻译区结构

翻译面板保留现有布局，输入区改为双按钮：

- 文本输入框
- `翻译文本`
- `粘贴并翻译`

在 `粘贴并翻译` 按钮附近增加一条轻量说明文案：

- `读取系统剪贴板内容，不是当前 PDF 划选内容`

建议按钮语义：

- `翻译文本`：针对输入框已有内容
- `粘贴并翻译`：从系统剪贴板读取，再立即调用翻译

### 7.2 成功路径

用户点击 `粘贴并翻译` 后：

1. 前端在按钮点击回调内尝试读取剪贴板文本
2. 对读取结果做统一规范化：
   - 使用 `trim()` 结果判断是否为有效文本
   - `trim()` 后为空字符串时，视为“没有可翻译的文本”
3. 如果读取到有效文本：
   - 把 `trim()` 后的文本写入 `manualText`
   - 清除已有剪贴板错误提示
   - 按现有契约调用 `api.translate([text])`
   - 复用现有翻译逻辑发起翻译
4. 翻译结果继续沿用现有 `translations` 列表展示

### 7.3 失败路径

采用局部内联提示，不使用全局弹窗作为主反馈。

需要覆盖以下场景：

1. 剪贴板为空或只有空白字符
   - 文案：`剪贴板里没有可翻译的文本`

2. 浏览器拒绝读取剪贴板权限
   - 文案：`浏览器不允许直接读取剪贴板，请手动粘贴`

3. 当前环境不支持 `navigator.clipboard.readText`
   - 文案：`当前环境不支持一键读取剪贴板，请手动粘贴`

4. 读取剪贴板时出现其他未知异常
   - 文案：`读取剪贴板失败，请手动粘贴后再试`

5. 读取剪贴板成功，但后续翻译请求失败
   - 继续沿用当前翻译失败展示机制
   - 不要清空已经写入输入框的文本
   - `clipboardFeedback` 不额外叠加成功提示或错误提示，避免与现有翻译失败展示冲突

### 7.4 用户体验要求

- 错误提示应显示在翻译区内部，不能打断整页操作
- 读取剪贴板失败时，不要清空用户输入框里已有内容
- 读取成功后要保留输入框文本，便于用户复查或重试
- 点击 `粘贴并翻译` 后，应立即进入共享忙碌态，直到以下任一时刻结束：
  - 剪贴板读取失败并完成提示展示
  - 或翻译请求结束
- 共享忙碌态期间，`翻译文本` 与 `粘贴并翻译` 两个按钮都应禁用，避免重复触发
- 共享忙碌态期间，重复点击应直接忽略，不允许并发发起第二次剪贴板读取或翻译
- “粘贴并翻译”成功路径不展示单独的成功提示，只负责清除旧的剪贴板错误提示

## 8. 技术设计

### 8.1 状态

在 `PdfReader.vue` 增加一组局部状态即可：

- `clipboardFeedback` 或类似状态
  - 仅用于显示剪贴板相关失败/提示性文案
- 可选：`isClipboardLoading`
  - 如果希望把“读取剪贴板中”与“翻译中”区分开，可以加
  - 如果不希望增加状态复杂度，也可以直接并入 `isTranslating`

推荐最小实现：

- 不单独拆 `isClipboardLoading`
- 点击 `粘贴并翻译` 后立即进入统一忙碌态
- 统一忙碌态可继续沿用现有 `isTranslating`，但语义上覆盖“读取剪贴板 + 发起翻译”整个链路
- 只新增一条轻量反馈状态

### 8.2 反馈生命周期

`clipboardFeedback` 的清理规则明确为：

- 剪贴板成功读取并准备进入翻译时，立即清理旧反馈
- 用户手动修改翻译输入框内容时，清理旧反馈
- 用户再次点击 `翻译文本` 或 `粘贴并翻译` 时，先清理旧反馈，再执行后续逻辑
- 面板关闭时不要求单独持久化该状态，按组件现有生命周期自然重置即可

### 8.3 逻辑复用

不要把翻译逻辑复制两份。

建议拆分为两层：

1. 文本准备层
   - 手动输入路径：与剪贴板路径使用同一套“去首尾空白 + 空文本校验”规则
   - 剪贴板路径：读取剪贴板后写入 `manualText`

2. 翻译执行层
   - 单独抽一个基于文本参数执行翻译的函数
   - `translateSelected()` 与 `pasteAndTranslate()` 共用这一层

这样可以避免未来两条路径出现行为分叉。

补充约束：

- 手动输入路径与剪贴板路径都使用统一的文本规范化规则
- 两条路径最终都向共享翻译执行层传入“已经完成规范化的单条字符串”
- 本次不新增前端分段、分句或按换行拆分逻辑

### 8.4 浏览器能力约束

核心 API：

- `navigator.clipboard.readText()`

约束：

- 需要用户手势触发
- 通常要求安全上下文
- 某些环境会拒绝权限或根本不支持

因此实现必须满足：

- 只在按钮点击事件中调用
- 能力检测失败时回退到手动粘贴
- 永远保留手动输入框作为兜底

推荐映射规则：

- `typeof navigator === 'undefined'`、`!window.isSecureContext`、或 `navigator.clipboard.readText` 不存在：
  - 视为“不支持”
- `readText()` 抛出 `NotAllowedError` 或等价权限拒绝异常：
  - 视为“浏览器不允许直接读取剪贴板”
- 其他异常：
  - 走通用失败文案 `读取剪贴板失败，请手动粘贴后再试`

### 8.6 大文本边界

本次不新增前端字符数上限、截断策略或单独的大文本确认弹窗：

- 如果剪贴板文本很大，仍按现有翻译链路处理
- 若因文本过长触发后端或接口失败，继续沿用现有翻译失败展示
- 输入框内保留本次文本，便于用户自行裁剪后重试

### 8.5 文案与 i18n

本次范围内沿用当前翻译面板的中文本地文案风格：

- 可以直接使用中文文案
- 不在本次需求里顺手引入新的 i18n 改造

- 翻译区存在局部提示展示位置
- 翻译区存在“读取系统剪贴板内容，不是当前 PDF 划选内容”的提示文案或等价说明

   - 触发 `api.translate([text])`

2. 剪贴板为空
   - 不发起翻译
   - 显示局部错误提示

   - 不发起翻译
   - 显示回退提示

   - 不覆盖现有输入框内容

## 10. 风险与回退

### 风险

- 某些浏览器/部署环境下无法直接读取剪贴板
- 用户可能误以为这是“自动读取当前 PDF 选区”，但实际上读取的是系统剪贴板
- 一键触发意味着它会像现有手动翻译一样直接进入翻译链路，因此也会沿用现有配额与限流结果

### 回退策略

- 保留手动输入框和现有翻译按钮
- 一旦剪贴板能力失败，只提示“请手动粘贴”，不影响现有功能
- 如果后续用户反馈对“直接触发翻译”理解成本偏高，再单独评估是否增加说明文案强化或轻量确认，不在本次范围内提前实现

## 11. 验收标准

满足以下条件即可验收：

1. 翻译面板出现 `粘贴并翻译` 按钮
2. 按钮附近出现“读取系统剪贴板内容，不是当前 PDF 划选内容”的说明文案或等价提示
3. 点击后可读取剪贴板文本并发起翻译
4. 读取成功时，文本会以统一规范化规则写入输入框
5. 两个入口最终都按既有契约调用 `api.translate([text])`
6. 读取失败、剪贴板为空、权限不足或未知异常时，翻译区内有明确友好提示
7. 点击 `粘贴并翻译` 后立即进入共享忙碌态，忙碌期间两个按钮禁用，重复触发会被忽略
8. “粘贴并翻译”成功路径不展示额外成功提示，只清理旧的剪贴板错误提示
9. 手动粘贴 + `翻译文本` 的旧路径不受影响
10. 不新增后端接口改动
