# Frontend Long Conversation Performance Design

**Date:** 2026-03-31

## Scope

本设计只处理 `frontend-vue` 在单会话累计 10+ 轮问答后出现的前端卡顿、浏览器无响应、流式阶段掉帧等问题。

本设计覆盖：

- 主聊天区长会话渲染性能
- 流式输出期间的增量渲染性能
- 右侧问题大纲的更新策略
- 本地持久化对主线程的影响
- 长会话下的自动滚动行为

本设计不覆盖：

- `fastQA` / `highThinkingQA` 后端检索或生成耗时
- DOI 本身是否正确插入
- 引用定位算法本身
- 会话上下文裁剪策略
- 样式重构或 UI 改版

---

## Problem Statement

当前系统在一个对话中累计约 10 轮到十几轮问答后，前端会出现明显卡顿，严重时浏览器标签页直接无响应。用户已经确认：

- 后端仍可继续工作
- 新开会话通常恢复正常
- 问题更像是单个前端会话视图的累积开销，而不是单次请求失败

这说明瓶颈主要在前端长会话视图的响应式更新、HTML 渲染、局部状态计算与持久化开销叠加，而不是后端 ask 主链路。

---

## Evidence

以下证据已经从现有代码确认，不是猜测。

### 1. 消息列表仍然按整段会话全量渲染

文件：`frontend-vue/src/views/Home.vue`

当前模板直接对 `store.currentMessages` 做整表 `v-for`：

- 每次流式 chunk 到达后，整个消息列表仍会参与一次响应式更新
- 即使 `v-memo` 存在，也需要先生成 memo key
- 会话越长，单次更新涉及的消息对象越多

### 2. `v-memo` 依赖的 memo key 生成成本本身偏高

文件：`frontend-vue/src/utils/messageRenderMemo.js`

当前 `buildMessageRenderMemoKey()` 会对消息内容、steps、references、referenceLinks、doiLocations 整体构造对象后再 `JSON.stringify()`。

这意味着：

- 历史消息虽然内容不变，但每次父层渲染仍会走一次签名计算
- 长答案、长 steps、长引用列表会放大 CPU 成本
- `v-memo` 减少了 DOM patch，但没有减少所有 JS 侧的序列化工作

### 3. 右侧问题大纲对整个会话做全量扫描

文件：`frontend-vue/src/views/Home.vue`

`questionOutlineItems` 是基于 `store.currentMessages.reduce()` 的计算属性。

这意味着：

- 每次消息数组更新时，大纲都要重新遍历整段对话
- 流式输出期间，虽然新增内容通常只落在最后一条 assistant message，但用户问题列表仍在被重复计算

### 4. 流式消息目标定位仍会逆序扫描消息数组

文件：`frontend-vue/src/views/Home.vue`

`getStreamingTargetMessage()` 会从当前会话消息尾部逆序寻找目标 assistant message。

这意味着：

- 每个流式事件、step 更新、metadata 合并都可能再次扫描会话消息数组
- 在 20+ 消息后虽然单次开销不算灾难，但它和其他热点叠加后会持续放大

### 5. 自动滚动在流式期间持续触发布局相关工作

文件：`frontend-vue/src/views/Home.vue`

`scrollToBottom()` 每次都在 `requestAnimationFrame + nextTick` 后强制写 `scrollTop = scrollHeight`。

这意味着：

- 用户即使已经向上查看历史内容，也仍会被强制拉到底部
- 流式刷新期间会频繁读取/写入布局相关属性
- 长会话 DOM 越大，滚动相关的布局成本越高

### 6. 本地持久化仍会对整个 chats 状态做序列化

文件：`frontend-vue/src/stores/chatStore.js`

`saveChats()` 在流式期间虽已有 debounce，但最终仍会调用 `persistChatsNow()`，把 `chats` 清洗后整块写入 `localStorage`。

这意味着：

- 当单个对话很长时，`sanitizeChats()` 和 `JSON.stringify()` 仍会占用主线程
- 它不是当前唯一瓶颈，但会制造明显卡顿尖峰

## Goal

本设计的目标不是“理论上更优雅”，而是把真实用户体验拉回可用区间。

### 产品目标

1. 单会话问答达到 20+ 轮时，前端仍保持可交互，不出现浏览器无响应。
2. 流式输出期间页面保持平滑，不能在 DOI 或长 markdown 渲染阶段明显卡死。
3. 不牺牲当前功能：处理步骤、引用、查看原文、右侧问题大纲、聊天记录持久化都仍可用。
4. 不要求用户换新会话才能继续问。

### 性能目标

1. 流式阶段只允许最后一条活跃消息持续高频更新，历史消息必须尽量冻结。
2. 非必要的整会话扫描应从“每个 chunk 都发生”降低到“仅在结构变化时发生”。
3. 本地持久化不能继续以“每次流式阶段都序列化整段聊天”为默认路径。

### 验收目标

1. 同一会话连续进行 20 轮以上问答后，输入框、滚动、侧边栏点击仍能在可接受延迟内响应。
2. 单次流式输出期间，不再出现由于前端自身更新导致的长时间卡死。
3. 浏览器性能面板中，单次 chunk 刷新对应的脚本执行成本明显低于当前实现。

---

## Non-Goals

本设计不做以下事情：

1. 不引入服务端分页会话消息协议
2. 不修改后端返回的 ask / stream 数据格式
3. 不一次性上虚拟列表重构全部聊天视图
4. 不改变消息持久化 authority
5. 不为了性能删掉步骤面板、引用面板或 markdown 渲染

---

## Approaches

### 方案 A：只继续调小流式 flush 频率

做法：

- 增大前端流式渲染 throttle 间隔
- 减少 markdown 重渲染频率

优点：

- 改动最小
- 风险低

缺点：

- 只是在延迟重渲染，不解决历史消息、大纲、滚动、持久化等累积成本
- 会直接损伤流式输出的平滑度
- 达不到 20+ 轮稳定目标

结论：不推荐作为主方案。

### 方案 B：分阶段削减无意义更新成本

做法：

- 冻结历史消息的渲染依赖
- 让右侧大纲从“全量重算”改为“结构变更时更新”
- 缓存流式目标索引
- 自动滚动只在用户接近底部时触发
- 流式期间只渲染最近窗口，历史消息折叠
- 后续再决定是否上真正虚拟列表

优点：

- 对现有架构侵入适中
- 先打掉主要瓶颈，再决定是否需要更重的改造
- 风险和收益比最好

缺点：

- 需要引入“历史折叠/窗口渲染”逻辑
- 需要较仔细的回归验证

结论：推荐方案。

### 方案 C：直接上完整虚拟列表

做法：

- 把消息区改造成真正的 virtual list / window list
- 只渲染可视区和缓冲区 DOM

优点：

- 理论上性能上限最高

缺点：

- 与当前富文本、流式更新、步骤折叠、引用区、滚动定位耦合很深
- 改动面大，短期高风险
- 需要处理可变高度消息、流式增长、跳转到问题锚点等复杂细节

结论：作为 P2 预研，不作为第一阶段主方案。

## Recommended Design

采用方案 B，分三阶段落地。

### P0：停止长会话里的无效重复工作

这是第一优先级，目标是在不改变用户可见交互的前提下，立刻削减每个 chunk 的前端 CPU 成本。

#### P0-1. 冻结历史消息的 memo 计算

目标：

- 旧消息不再在每次父组件更新时重新构造完整 JSON 签名
- 只有当前正在流式更新的那条 assistant message 使用动态 key

设计：

- 为消息引入轻量 render version / stable render signature 概念
- 历史完成态消息只在真正发生内容变化时刷新 signature
- 对于 `isComplete=true` 且不再变动的消息，避免继续遍历 `steps` / `references` / `doiLocations` 做深序列化

约束：

- 不能破坏步骤展开收起、引用位置展示等 UI 状态更新
- 需要把“内容变更”和“纯 UI 状态变更”分开建模

#### P0-2. 缓存当前流式目标消息位置

目标：

- 避免每个流式事件都去扫描整段消息列表

设计：

- 在开始流式时记录 `streamingChatId + requestId + targetMessageIndex`
- 如果消息数组结构未变化，则直接按索引命中
- 只有在会话切换、消息插入、流式异常恢复时才回退到扫描逻辑

#### P0-3. 右侧问题大纲改成结构化更新

目标：

- assistant 流式输出时，不再因为最后一条答案增长而重算全部用户问题列表

设计：

- 问题大纲只在以下事件更新：
  - 新增 user message
  - 删除消息
  - 切换会话
  - 首次加载历史消息
- assistant message 内容变化不触发大纲重建

#### P0-4. 自动滚动增加 near-bottom 保护

目标：

- 用户不在底部时，不再强制滚动
- 减少流式阶段不必要的布局写入

设计：

- 维护 `isNearBottom` 状态
- 建议首版 near-bottom 阈值取 `120px`
- 仅当用户本来就在底部附近时自动跟随最新内容
- 用户上滑阅读历史后，停止自动滚动，并提供“回到底部”提示或复用现有行为

#### P0-5. 流式期间降低本地持久化频率

目标：

- 减少大 JSON 写入 localStorage 造成的主线程尖峰

设计：

- 流式进行中只做更粗粒度 debounce
- `done`、`error`、`abort` 时强制落一次
- 非流式修改仍保持及时持久化

默认建议：

- 流式期间 debounce 首版固定为 `1200ms`
- 回答完成时立即 `force persist`

### P1：限制流式阶段的可见渲染窗口

这是第二优先级，目标是让长会话期间 DOM 规模和每次 patch 范围受控。

#### P1-1. 流式期间只渲染最近窗口

设计：

- 当会话消息数超过阈值时，聊天区默认只渲染最近 `N` 条消息
- 首版启用阈值建议为 `totalMessages > 30`
- 建议首版 `N = 24`
- 更早消息折叠成一个“查看更早消息”块

理由：

- 问题不是数据不能保存，而是没有必要在流式阶段把全部历史 DOM 都挂在页面上
- 这一步能显著降低长会话的重排和 patch 成本

#### P1-2. 旧消息展开采用分段恢复，不一次性全放出

设计：

- 用户点击“查看更早消息”后，按批次展开，例如每次 `20` 条
- 展开动作只影响历史区，不影响活跃流式区

#### P1-3. 右侧大纲与主消息区解耦

设计：

- 即使主消息区只显示最近窗口，右侧问题大纲仍可展示完整问题索引
- 点击大纲定位到已折叠历史问题时，先自动展开到包含该问题的批次，再滚动定位

### P2：预留真正虚拟列表能力

这是第三优先级，不作为当前立刻实施内容，但要在设计上预留边界。

适用条件：

- P0 + P1 完成后，20+ 轮仍出现明显卡顿
- 或未来单对话消息规模达到 100+ 条

预留原则：

- 不把消息组件与列表实现强耦合
- 消息高度测量逻辑单独封装
- 问题锚点、引用跳转、流式增长要提前留出接口

## Data And State Design

### 1. 消息渲染状态分层

需要把“消息业务内容”和“消息渲染控制状态”分开。

建议引入两层概念：

- 消息业务数据：`content / steps / references / doiLocations / isComplete`
- 渲染控制数据：`renderVersion / renderFrozen / isStreamingTarget`

原则：

- 业务数据变更才允许刷新 heavy render signature
- 纯 UI 状态如 `stepsCollapsed` 不应重新计算完整 markdown / 引用签名
- 渲染控制状态优先放在组件内存态或 `WeakMap`，不写回持久化消息体
- `localStorage` / `public-service` / 后端消息协议中不新增这些前端临时字段

### 2. 大纲状态独立维护

当前 `questionOutlineItems` 是“从消息列表推导”。

建议改为：

- 在消息插入 user turn 时增量维护 outline state
- 会话切换时从历史消息重建一次
- 流式 assistant 更新不触碰 outline state

### 3. 可见消息窗口状态

建议引入：

- `visibleMessageStart`
- `visibleMessageEnd`
- `hasHiddenOlderMessages`
- `hiddenMessageCount`

窗口化实现必须同时引入稳定消息身份映射：

- 每条渲染消息必须携带 `absoluteMessageIndex`
- 如果后续已有稳定 `messageId`，则可以用 `messageId` 作为主身份，但首版至少要保留 `absoluteMessageIndex`
- `visibleMessages` 只是完整消息数组的视图切片，不能把窗口内局部 index 当成业务身份

要求：

- 这些状态只控制渲染，不改变真实消息数组
- 聊天记录持久化、引用位置、消息同步都仍基于完整消息数据
- 以下入口统一基于稳定身份，而不是窗口局部 index：
  - `v-for key`
  - DOM `data-message-index`
  - 右侧问题大纲项
  - steps 展开/收起
  - DOI 引用点击与查看原文
  - 高亮定位与滚动定位
- 当目标消息位于隐藏历史区时，必须先展开包含该消息的批次，再执行定位、点击或引用解析

### 4. 自动滚动状态

建议引入：

- `isNearBottom`
- `lastManualScrollAt`
- `pendingAutoScroll`

原则：

- 自动滚动必须尊重用户手动浏览历史的行为
- 只有当前消息是活跃流式目标，且用户处于 near-bottom，才允许自动滚动

### 5. Runtime-Only 状态清单

以下状态只能存在于组件运行时内存态、`WeakMap`、局部 `ref` 或其他非持久化容器中：

- `streamRequestId`
- `streamingChatId`
- `targetMessageIndex`
- `renderVersion`
- `renderFrozen`
- `isStreamingTarget`
- `highlightedQuestionMessageIndex`
- `isNearBottom`
- `lastManualScrollAt`
- `pendingAutoScroll`
- 历史窗口当前展开位置

约束：

- 不允许把这些字段写进 `store.chats` 的消息体持久化结构
- 不允许把这些字段透传给 `public-service` 或后端消息协议
- 如果实现中不得不暂存在消息对象上，则必须在持久化前显式清洗

## UX Rules

### 1. 长会话默认优先保证输入和流式顺滑

在消息很多时，优先级如下：

1. 输入框可响应
2. 流式输出平滑
3. 最新几轮消息完整可见
4. 更早历史可按需展开

历史消息不能为了“始终全部显示”而拖垮整个页面。

### 2. 折叠历史消息不改变会话事实

用户需要明确知道：

- 历史消息没有丢
- 只是当前为了性能先折叠显示
- 展开后仍可查看原始答案、引用、步骤

### 3. 大纲必须继续可用

因为长会话下，右侧问题大纲本身就是重要导航工具。

所以：

- 不能因为主消息区折叠历史，就把大纲也裁掉
- 大纲需要成为访问折叠历史的重要入口

### 4. 刷新恢复必须有明确边界

刷新后行为不能靠实现时临时决定，必须遵循以下矩阵：

必须恢复：

- 消息内容 `content`
- 引用 `references`
- 引用链接 `referenceLinks`
- 引用位置 `doiLocations`
- 已完成消息的 `steps`
- 对话本身的完整消息数组

可以继续恢复：

- `stepsCollapsed`

固定规则：

- `stepsCollapsed` 继续沿用当前行为，允许持久化并在刷新后恢复
- 该字段虽然属于 UI 状态，但不参与 heavy render signature 的深计算
- 历史窗口状态不跟随 `stepsCollapsed` 一起持久化，二者必须分离

必须重置：

- 历史窗口展开位置
- 高亮状态
- `isNearBottom`
- `lastManualScrollAt`
- `pendingAutoScroll`
- `streamRequestId`
- `streamingChatId`
- `targetMessageIndex`

流式中途刷新约束：

- 如果刷新发生在未完成回答期间，该回答按“未完成但可恢复查看的静态内容”处理
- 刷新后不自动恢复原流式连接
- 刷新后允许继续提问，但不能把旧的 runtime stream 状态误判为仍在生成

## Logging And Measurement

本设计建议补充最小必要的前端性能观测点，但不引入复杂埋点系统。

建议记录：

- 单次流式 flush 的耗时
- 当前消息总数
- 当前渲染窗口消息数
- 是否处于 auto-scroll
- 本次 localStorage persist 的消息规模
- 当前回答对应的 `saveChats()` 触发次数

用途：

- 验证 P0 / P1 是否真的生效
- 识别仍然存在的尖峰位置

约束：

- 只在开发环境开启详细日志
- 不能把整段消息内容打进控制台

建议增加一条实现约束：

- 性能日志必须可整体关闭，避免调试埋点本身再次放大长会话成本

## Risks

### 1. 折叠历史消息可能影响用户对“上下文连续性”的感知

缓解：

- 默认保留最近窗口足够大
- 折叠区明确展示“已折叠 X 条更早消息”
- 点击大纲时自动展开相关区间

### 2. 过度缓存 render key 可能导致 UI 不刷新

缓解：

- 把内容变更触发点和 UI 状态触发点列成明确规则
- 先对 streaming / done / steps toggle / doiLocations 更新做覆盖测试

### 3. 调高持久化 debounce 可能增加浏览器意外关闭时的少量丢失窗口

缓解：

- `done` / `error` / `abort` 强制持久化
- 对关键动作保留显式落盘

### 4. 历史折叠可能影响“查找页面文本”体验

缓解：

- 这是可接受折中，优先保证页面可用
- 后续若需要，再通过虚拟列表或搜索入口增强

### 5. 新增渲染窗口逻辑可能影响引用跳转与问题定位

缓解：

- 所有定位入口统一先经过“确保目标消息可见”的展开逻辑
- `PdfReader` 引用位置跳转与右侧问题大纲跳转都必须覆盖回归验证

### 6. 运行时流式状态误入持久化结构会导致刷新恢复异常

缓解：

- 在实现阶段把 runtime-only 字段做成单独清单
- 持久化前统一清洗
- 为“刷新后不误判仍在流式中”增加专项验证

---

## Acceptance Criteria

### 功能验收

1. 同一会话达到 20+ 轮问答后，前端仍可继续提问、停止生成、切换右侧大纲、查看最近回答。
2. 流式过程中，用户若上滑查看历史，页面不再强制抢滚动焦点。
3. 历史折叠后，用户仍可通过显式入口展开并查看旧消息。
4. 点击右侧问题大纲定位旧问题时，能够正确展开并定位。

### 性能验收

1. 使用固定的长会话 workload 录制浏览器性能 trace，`20` 轮场景下 `p95 chunk script time` 相比改造前有明确下降。
2. 同一 workload 下，流式期间 `saveChats()` 触发频率必须显著低于“每个 chunk 一次”，并符合实现设定的 debounce 预期。
3. 在消息总数超过 `30` 条后，实际渲染条数仍保持在窗口上限附近，而不是再次回到全量挂载。
4. `30` 轮场景下页面仍可继续输入、滚动和点击大纲，且不出现浏览器标签页无响应。

### 回归验收

1. Markdown、表格、DOI 链接、引用位置、步骤面板仍正常显示。
2. 聊天记录刷新后仍能恢复。
3. 新会话、短会话体验不退化。
4. 大纲跳转、DOI 引用点击、steps 展开在历史窗口开启后仍定位到正确消息。

## Implementation Order

### Task 1: P0 基础削峰

- 冻结历史消息 memo 计算
- 缓存流式目标消息索引
- 大纲从全量 reduce 改成结构化更新
- 自动滚动加 near-bottom 保护
- 提高流式持久化 debounce

### Task 2: P0 验证

- 验证功能不回退
- 评估是否还需要 P1

### Task 3: P1 历史折叠窗口

- 只渲染最近窗口
- 支持按批次展开旧消息
- 接通右侧大纲到折叠历史的展开定位

### Task 4: P1 验证

- 验证 20+ 轮场景是否已稳定
- 如果仍存在明显卡顿，再启动 P2 预研

### Task 5: P2 预研

- 评估虚拟列表是否值得引入
- 输出独立预研结论，不直接重构主视图

## Test Plan

### 手工验证

1. 单会话连续问答 20 轮以上，观察输入、滚动、折叠、展开是否仍流畅。
2. 长回答中包含 markdown 表格、DOI、引用位置、步骤面板，观察流式阶段是否掉帧。
3. 流式时主动向上滚动，确认页面不会继续抢滚动。
4. 刷新页面，确认聊天记录恢复正常。

### 开发验证

1. 在开发环境记录每次 flush 的渲染耗时。
2. 对比改造前后的 `currentMessages.length` 与实际渲染条数。
3. 对比 `saveChats()` 的调用频率和单次持久化体积。
4. 使用同一组固定 workload：
   - 至少包含 30 轮消息
   - 至少包含 markdown 表格
   - 至少包含 DOI 链接
   - 至少包含步骤面板更新
   - 至少包含一次用户上滑查看历史后继续流式输出

### 回归重点

1. 右侧问题大纲跳转
2. PDF 引用位置展示
3. DOI 链接点击
4. 流式中断与错误态展示
5. 刷新恢复后 steps / 引用 / 消息数量正确
6. 流式中途刷新后不会误判仍在生成

## Open Questions

1. 历史折叠窗口的默认阈值取 `20`、`24` 还是 `30` 更合适，需要首轮验证数据决定。
2. 如果后续引入真正虚拟列表，当前消息组件是否需要进一步拆分为更细粒度子组件。

---

## Rollout Guard

为避免一次性切换影响线上可用性，建议实现时预留以下回退机制：

1. P0 与 P1 分开提交和验证，不合并成一个大改动。
2. 历史窗口化逻辑保留单独开关，便于快速回退到“全量显示”。
3. 如果 P0 落地后 20+ 轮场景已经稳定，P1 可以延后，不必强推。

---

## Conclusion

当前前端卡顿的核心不是单一函数慢，而是“长会话下多处按全量消息工作”的累积结果。

最合理的路线不是直接上重型虚拟列表，而是先用 P0 打掉每个 chunk 的无效重复工作，再用 P1 控制可见 DOM 窗口，最后视结果决定是否做 P2。

这个顺序能在风险、收益和落地速度之间取得最好平衡。
