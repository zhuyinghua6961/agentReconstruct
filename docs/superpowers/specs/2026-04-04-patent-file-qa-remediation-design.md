# Patent File QA Remediation Design

**Date:** 2026-04-04

## Scope

本设计定义 `patent` 模式下文件 QA / 混合 QA 的修复方案，目标是把当前“能路由但答案质量和行为明显偏离 `fastQA`”的问题一次性收敛掉。

本设计覆盖：

1. `patent` PDF 文件 QA 对齐 `fastQA` 的 prompt 规则
2. `patent` 混合 QA 对齐 `fastQA` 的统一证据合成思路
3. 多 PDF 场景，尤其是“对比两篇文献”场景的内容组织与截断策略
4. `patent` 文件 QA / 混合 QA 的流式输出、步骤展示、答案格式
5. 面向实现和验收的失败样例、验收样例、测试要求

本设计不覆盖：

1. 前端页面、按钮、入口路径改版
2. `gateway` 的文件判定逻辑重写
3. `patent` 普通 QA 主链路的检索与回答逻辑重构
4. `highThinkingQA` 的文件执行能力扩容
5. `patent` 知识库底层召回算法改造

---

## Goal

建立一套真正可用的 `patent` 文件 QA / 混合 QA 行为，使其在产品语义上与 `fastQA` 保持一致，而不是只完成“路由到了 patent 服务”这一层壳子。

最终目标：

1. `patent` 文件 QA 的回答质量、格式、步骤和流式体验与 `fastQA` 同级
2. `patent` 混合 QA 采用统一证据合成，不再输出拼接式占位答案
3. “对比两篇文献”这类多 PDF 问题能够稳定看到两篇文献都参与回答
4. `patent` 与 `fastQA` 在文件 QA / 混合 QA 领域共享同一套核心规则，只保留 `mode`、知识库和普通 QA 下游差异
5. 修复结果可通过明确验收样例和自动化测试证明，而不是人工感觉“像是好了”

---

## Problem Statement

当前 `patent` 已经可以承接文件 QA / 混合 QA 请求，但这条链路仍然存在明显缺陷，属于“通了路由，没有真正对齐能力”。

已确认的问题如下。

### 1. 文件 QA prompt 没有复用 `fastQA` 的强约束规则

`fastQA` 的 PDF QA prompt 明确要求：

1. 严格基于 PDF 原文
2. 禁止使用通用知识补全文档未写内容
3. `summary` 与普通问答分支分离
4. 带 KB 时只允许验证，不允许拿 KB 补正文
5. 输出格式和失败文案都有清晰约束

而 `patent` 当前 PDF QA prompt 更像轻量摘要提示，缺少这些关键约束，导致：

1. 容易输出偏泛化的总结
2. 容易把用户的比较问题回答成单篇总结
3. 和 `fastQA` 的答案风格、边界和可靠性明显不一致

### 2. 多 PDF 内容虽然被读取，但没有按“多文献问答”组织

`patent` 当前会把多篇 PDF 正文拼接起来，但没有在 prompt 中明确声明：

1. 当前有几篇文献
2. 每篇文献的边界和标识
3. 当前问题是否要求比较
4. 当模型做总结时应该先分篇，再做归纳

这会导致多文献问题被模型误判成单文档摘要。

### 3. 多 PDF 内容会被整体截断，后面的文献可能直接丢失

当前 `patent` 会先对每篇 PDF 各自截断，再把所有 PDF 拼接成一个大文本，随后又对整个大文本统一截断到固定字符数。

这会导致：

1. 第一篇文献占满预算
2. 第二篇或后续文献被整体裁掉
3. 用户问“对比两篇文献”，模型实际只看到了第一篇

这不是回答质量波动，而是输入数据结构本身就不满足需求。

### 4. 混合 QA 仍然是分段拼接，不是统一合成

当前 `patent` 混合 QA 会先分别生成：

1. `PDF 部分`
2. `表格部分`

然后再拼出一个总答案，并附一句“知识库结果会在下游继续合并”。

这和 `fastQA` 当前的“PDF 证据 + 表格证据 + KB 证据统一合成一个答案”的逻辑不同，也会导致：

1. 最终答案割裂
2. KB 不真正参与文件答案生成
3. 文件问题看起来像占位式拼接，不像真正完成了推理

### 5. fallback 行为会把失败退化成占位式摘要

当前 `patent` PDF QA fallback 会产出：

`基于 PDF 原文提取，文档要点如下：`

这种 fallback 对单篇摘要还勉强可接受，但对比较、问答、混合证据场景是不合格的。它会掩盖主链路失败，并给人一种“返回了答案，但其实没完成任务”的假象。

---

## Non-Goals

本设计明确不做以下事情：

1. 不在 `patent` 再写第二套文件判定器
2. 不为 `patent` 新造一套弱化版 prompt
3. 不为了追求快速交付保留“PDF 部分 / 表格部分 / 下游继续合并”这类拼接式占位输出
4. 不改前端按钮与主页面分流
5. 不动 `patent` 普通 QA 主链路已在进行中的独立开发内容

---

## Design Summary

本设计采用“核心链路收敛”的方案，而不是局部打补丁。

### 1. `patent` 文件 QA / 混合 QA 向 `fastQA` 核心行为全面对齐

对齐范围包括：

1. PDF QA prompt 规则
2. 多 PDF 内容组织格式
3. 多 PDF 截断策略
4. 混合 QA 的统一证据合成方式
5. 流式输出行为
6. steps 展示结构
7. 失败时的可解释文案

这不是“语义上参考 `fastQA`”，而是要求 `patent` 调用或镜像以下明确合同：

1. PDF QA prompt 合同：`fastQA/app/modules/qa_pdf/prompting.py`
2. PDF QA 截断合同：`fastQA/app/modules/qa_pdf/truncation.py`
3. PDF QA 运行时合同：`fastQA/app/modules/qa_pdf/engine.py`
4. 混合 QA 统一证据合成输入合同：`fastQA/app/modules/qa_tabular/service.py`

允许的实现方式只有两类：

1. 直接抽共享 builder / helper，由 `fastQA` 与 `patent` 共用
2. 保留独立实现，但必须用 parity snapshot / contract tests 锁定和上述合同一致

不接受“人工照着抄一版，看起来差不多”的做法。

### 2. `patent` 只保留 mode 级差异，不保留文件 QA 规则分叉

`patent` 与 `fastQA` 允许保留的差异只有：

1. `requested_mode=patent`
2. `patent` 自己的知识库
3. 文件部分结束后接回 `patent` 自己的普通 QA / 下游链路
4. 如有需要，system identity 的轻微文案差异

除此之外，不再保留两套不同的文件 QA 规则。

### 3. “多文献比较”是一级场景，不是摘要场景的副产物

当用户问题包含显式或隐式比较意图时，系统必须进入 compare-aware 分支，而不是继续沿用“请总结这份 PDF”的摘要 prompt。

比较意图不限于“对比 / 比较 / compare / versus / vs”，还至少应覆盖：

1. “异同 / 有什么不同 / 有什么相同”
2. “哪篇更好 / 哪篇效果更好 / 谁优谁劣”
3. “分别讲了什么”
4. “放在一起看 / 放在一起比较”
5. “两个方法有什么区别”
6. “这两篇的结论一致吗”

如果用户已经显式选中了多篇 PDF，且问题明显要求跨文献归纳或优劣判断，也必须进入 compare-aware 分支。

这个分支必须要求模型：

1. 明确识别有多篇文献
2. 分别归纳每篇文献
3. 输出相同点和差异点
4. 如果某篇文献证据不足，要显式指出

## Root Cause Analysis

### Root Cause 1: `patent` 当前 PDF QA prompt 是单文档摘要导向

当前 prompt 只有“总结、主题、方法、结果、结论”这类摘要指令，没有比较导向的结构要求，也没有 `fastQA` 那种严格边界约束。

因此模型在多文献问题上会自然退化成“抓住一篇做总结”。

### Root Cause 2: 多文献输入没有被显式建模为多文档

虽然所有选中 PDF 的内容会被读出，但当前 prompt 对模型暴露的信息仍然接近“这是一个大文本块”，没有告诉模型每篇文献的独立身份和边界。

因此模型无法稳定地执行“逐篇对比”。

### Root Cause 3: 统一硬截断破坏了后置文献可见性

多文献拼接后统一截断，会系统性偏向前面的文献。

这会让“对比两篇文献”的问题在输入层就失真，即便 prompt 写对了也可能失效。

### Root Cause 4: 混合 QA 当前没有统一合成步骤

分段拼接答案无法表达跨来源证据之间的真实关系，也无法像 `fastQA` 那样生成统一结论。

### Root Cause 5: fallback 设计掩盖了主链路失败

抽取式 fallback 让用户看起来拿到了“一个答案”，但实际上系统没有完成用户要求的任务类型。

## Core Principles

## 1. 文件 QA 规则只保留一套强规则

`patent` 不再维护一套弱化版文件 QA prompt。对 PDF 原文约束、KB 使用边界、答案格式要求，应直接复用 `fastQA` 规则。

若实现上无法直接共享模块，则必须通过 snapshot / contract tests 证明：

1. 相同输入下，`patent` 与 `fastQA` 的 prompt 结构一致
2. 多文献截断结构一致
3. 混合 QA 统一合成输入字段一致

## 2. 多文献输入必须显式分篇

每篇 PDF 的文件名、正文边界、顺序都必须被明确暴露给模型。不能把多篇文献压成一个匿名文本块。

## 3. 多文献截断必须均衡，不允许“第一篇吃满预算”

只要问题依赖多篇文献，截断策略就必须保证每篇文献都保留最小可用上下文。

## 4. 混合 QA 必须统一合成，不能分段拼接冒充完成

文件证据和 KB 证据应共同进入最终答案生成，而不是先做一个文件半成品再追加说明“KB 下游继续合并”。

## 5. 比较问题必须有比较结构

用户问的是比较，答案就必须有比较结构。返回单篇总结，即使内容属实，也算功能失败。

## 6. 验收以可证明行为为准，不以主观感觉为准

修复完成必须通过：

1. 明确验收样例
2. steps 和 streaming 可见性
3. 自动化测试

---

## Required Changes

### A. Prompt Reuse and Prompt Structure

#### A1. 复用 `fastQA` 的 PDF QA system prompt 与主 user prompt 规则

`patent` 的 PDF 文件 QA 不再维持当前轻量 prompt，改为复用 `fastQA` 的核心规则：

1. 严格基于 PDF 原文
2. 禁止用通用知识补 PDF 未写内容
3. `summary` 问题与普通问答分支分离
4. 带 KB 时明确“只验证，不补充正文”
5. 输出结构化、可读、可解释

允许参数化的内容：

1. system identity 的少量 mode 文案
2. `mode=patent` 的日志 / metadata 字段

不允许参数化弱化的内容：

1. 原文约束
2. KB 边界
3. 输出结构要求
4. 失败说明标准

为了防止“名义复用，实际复制漂移”，本项还要求满足以下至少一条：

1. `patent` 直接调用 `fastQA` 的 prompt builder
2. 抽公共 prompt builder 给两边共用
3. 为 `patent` 新实现补充 parity snapshot tests，逐段比对 prompt 模板骨架、KB 约束块、summary 与非 summary 分支

#### A2. 新增 compare-aware prompt 分支

当问题命中比较意图时，应进入 compare-aware prompt，而不是普通 summary prompt。

比较意图识别必须同时覆盖：

1. 显式比较词
2. 隐式异同类问题
3. 多篇文献上的优劣判断
4. 要求分别说明各篇内容的请求
5. 负例排除：多篇 PDF 已上传，但问题只问其中一篇具体内容时，不应误入 compare 分支

比较 prompt 的最低要求：

1. 列出参与比较的文献数和文件名
2. 先分别总结每篇文献
3. 再按以下维度比较：
   - 研究主题 / 目标
   - 方法 / 技术路线
   - 核心结果 / 证据
   - 结论 / 贡献
4. 最后输出相同点、差异点和总体结论
5. 如果某篇文献内容不足，明确说明证据不足，不允许装作比较完成

#### A3. 混合 QA 复用 `fastQA` 的统一证据合成 prompt 约定

`patent` 混合 QA 不再采用“PDF 部分 + 表格部分 + 下游说明”的拼接式文案。

应复用 `fastQA` 当前统一答案合成的输入约定，把以下证据在单次生成中一起交给模型：

1. 表格执行结果
2. PDF 证据上下文
3. KB 证据上下文
4. KB 引用说明
5. `source_scope`

同时必须定义清楚来源优先级与冲突规则：

1. PDF 明确写出的事实，以 PDF 为主
2. 表格执行得出的结构化结果，以表格执行结果为主
3. KB 在文件 QA / 混合 QA 中只可做验证、补充出处或提供文件外背景标签，不可伪造文件中不存在的具体结论
4. 当 PDF / 表格 与 KB 冲突时，答案必须显式说明冲突，不允许静默用 KB 覆盖文件证据
5. 当 PDF 对某点沉默时，不允许把 KB 内容包装成“PDF 结论”

---

### B. Multi-PDF Input Organization

#### B1. 多篇 PDF 内容必须带清晰分隔头

多文献内容组织要改成可被统一截断器和模型稳定识别的结构，例如每篇文献都具备明确头部与正文块，而不是简单 `文件: xxx` 顺序拼接。

要求：

1. 每篇文献必须带文件名
2. 每篇文献必须带稳定分隔头
3. 多篇文献顺序必须与 `selected_execution_files` 保持一致
4. prompt 中必须显式说明当前共选中了多少篇文献

#### B2. 不再只传第一篇作为 `primary file`

当前单数文件名字段会误导模型认为这是单文档问答。

改造后：

1. prompt 中列出所有参与文件
2. metadata 中保留全部 `used_files`
3. 如仍需 `primary_file_id` 兼容字段，只作为兼容元数据，不作为模型主输入语义

---

### C. Multi-PDF Truncation

#### C1. 复用 `fastQA` 的 multi-doc 均衡截断策略

`patent` 不再对合并后的多文献大文本做一次整体前向截断。

改为：

1. 检测多文献结构
2. 按文献数均衡分配预算
3. 每篇文献保留边界完整的摘录
4. 输出截断说明，便于日志与调试

最低约束：

1. 每篇文献都必须保留文件标识
2. 每篇文献都必须至少保留一段摘要/引言类上下文
3. 每篇文献都必须至少保留一段结果、讨论或结论类上下文
4. 发送到模型前的最终文本必须可证明包含全部参与比较的文献标识

若预算不足以满足以上最低约束，则不得假装完成比较，必须进入显式失败路径。

#### C2. compare 场景的截断优先级

对比较问题，截断目标不是“尽量保留第一篇更多内容”，而是“保证每篇至少有足够上下文用于比较”。

因此 compare 场景下必须优先满足：

1. 每篇文献都有摘要级上下文
2. 每篇文献都有结果 / 结论级上下文
3. 不能出现一篇内容完整、另一篇完全缺失的情况

在 compare 场景下，低价值前置内容的优先级低于比较相关证据。至少应优先保留：

1. 研究目标或主题
2. 方法 / 技术路线
3. 关键结果或实验观察
4. 结论 / 贡献

作者信息、版权页、长篇前置说明等低价值前文，不得挤占掉第二篇文献的关键比较预算。

---

### D. Streaming and Steps

#### D1. 文件 QA 与混合 QA 都必须维持流式输出

`patent` 文件 QA / 混合 QA 的回答过程应与当前 `fastQA` 一致：

1. 能逐块返回答案正文
2. 能先发 metadata / steps，再持续输出内容
3. 前端不需要等待整段文本生成完才显示

补充约束：

1. 对单篇 PDF 问答、或底层回答器本身可安全逐块输出的场景，应真实逐块流出正文，不应把整段答案完全缓冲到结束后再统一下发。
2. 对 compare 场景，如果中途失败会导致前端看到“半截比较结论”这类误导性正文，则允许先持续输出 metadata / steps，并在比较结果确认成功后再按 chunk 下发最终正文。
3. 对 `pdf+kb` / `pdf+table+kb` 这类统一合成路径，如果最终答案依赖本地证据合成而不是下游逐 token 生成，则要求：
   - 不阻塞 metadata / steps 的流式可见性
   - 在进入最终 success terminal 之前按 chunk 下发合成后的最终正文
   - 不允许退回 `Patent KB participation:` / `PDF 部分：` / `表格部分：` 这类壳式输出

事件时序要求：

1. 先发送 metadata / 已知 steps
2. 再发送正文 chunks
3. 若中途失败，必须先补齐失败 step，再发 terminal 事件
4. 不允许在判定比较失败后继续流出看似完整的比较答案正文

#### D2. steps 必须覆盖关键阶段

最少应包含：

1. `dispatch`
2. `context_ready`
3. `pdf_extract`
4. `pdf_answer`
5. compare 场景下的 `multi_pdf_compare` 或等价步骤
6. 混合 QA 的证据准备步骤
7. 最终合成步骤

要求：

1. `running -> success/error` 状态完整
2. steps 同时出现在流式和非流式响应中
3. `metadata.steps` 与顶层 `steps` 一致
4. 流式结束后最终 step 状态与非流式聚合状态一致

#### D3. compare 场景的可解释步骤

当问题是多文献比较时，步骤里应明确显示系统识别到了比较场景，例如：

1. 已识别多文献比较请求
2. 已加载 N 篇文献证据
3. 已进入比较型答案生成

这样前端和调试侧都能看到系统是否真正走到了比较链路。

如果 compare 失败，必须至少有一个失败 step 明确指出失败原因，例如：

1. 某篇文献正文为空
2. compare 截断预算不足
3. 统一合成失败

---

### E. Answer Format

#### E1. PDF compare 场景的标准答案结构

当用户问题是“对比两篇文献”时，答案最低应满足以下结构：

1. 文献 1 概要
2. 文献 2 概要
3. 相同点
4. 差异点
5. 总结

如果实际参与文献大于两篇，则结构应扩展到对应篇数，但仍保留比较和总结部分。

这不是纯格式要求。每篇文献概要中都必须包含至少一个能和另一篇区分开的证据性事实，不允许只写空泛标签。

#### E2. 非 compare 的 PDF 文件 QA 仍沿用 `fastQA` 风格

对单篇总结、定向问答等场景，仍复用 `fastQA` 的文件 QA 格式，不再输出当前 `patent` 这类轻量摘要式固定模板。

#### E3. 混合 QA 输出必须是统一答案

混合 QA 最终返回的是一个合成后的完整答案，而不是：

1. `PDF 部分`
2. `表格部分`
3. `知识库结果会在下游继续合并`

这种占位式拼接输出必须被移除。

---

### F. Fallback Behavior

#### F1. 不允许用抽取式摘要冒充 compare 成功

如果 compare 主链路失败：

1. 不能退化成单篇总结
2. 不能返回“文档要点如下”这类与用户任务不匹配的内容
3. 必须明确说明当前无法完成比较，并指出缺失条件
4. 已经流出的正文若会误导前端显示为“比较已完成”，则必须通过失败终态覆盖该轮状态

#### F2. fallback 只能作为可解释错误，不是伪答案

可接受的 fallback 行为：

1. 文件不可读
2. 某篇 PDF 正文为空
3. 多篇文献中只有一篇提取成功
4. LLM 失败后返回带原因的任务失败说明

不可接受的 fallback 行为：

1. 用一篇文献代替多篇比较
2. 用纯抽取句子冒充完成复杂问答
3. 隐藏失败原因

## Implementation Boundaries

### 可以改动的区域

1. `patent` 文件 QA / 混合 QA 执行链路
2. `patent` 的 PDF prompt 构造逻辑
3. `patent` 多 PDF 文本组织与截断逻辑
4. `patent` 文件 / 混合 QA 的 streaming 与 steps 处理
5. 如有必要，抽共享模块供 `fastQA` / `patent` 复用

### 不应改动的区域

1. 前端主页面与三个 mode 按钮
2. `gateway` 文件判定器
3. `patent` 普通 QA 主链路的独立功能开发范围
4. `highThinkingQA` 普通 QA 行为

## Acceptance Scenarios

### Scenario 1: Patent 单篇 PDF 总结

输入：

1. `mode=patent`
2. 上传 1 篇 PDF
3. 提问“总结这篇文献的主要内容”

验收：

1. 回答基于 PDF 原文
2. 不引入 PDF 未写的背景知识
3. 有流式输出
4. steps 可见

### Scenario 2: Patent 两篇 PDF 比较

输入：

1. `mode=patent`
2. 上传并选中 2 篇 PDF
3. 提问“对比一下这两篇文献的内容”

验收：

1. 答案中明确出现两篇文献
2. 结构包含“各自概要 + 相同点 + 差异点 + 总结”
3. 每篇概要都至少包含一个来自对应文献、且能与另一篇区分开的事实性信息
4. 不能只总结第一篇，也不能用同一组结论改写后套到两篇上
5. steps 中可见比较场景识别和多文献加载
6. 若第二篇正文为空，答案必须说明无法完成完整比较
7. 测试数据必须使用 adversarial fixtures：两篇文献在主题、方法或结论上至少有一个明确差异，避免“随便写一段通用总结也像对了”

### Scenario 3: Patent PDF + KB 混合问答

输入：

1. `mode=patent`
2. 上传 1 篇 PDF
3. 提问“结合这篇 PDF 和知识库内容回答……”

验收：

1. 文件内容和 KB 内容进入同一条答案合成链路
2. 最终输出是统一答案
3. 不能返回“文件部分已生成，KB 下游继续合并”

### Scenario 4: Patent PDF + Table + KB 混合问答

输入：

1. `mode=patent`
2. 同时选中 PDF 和表格
3. 用户显式要求结合文件与知识库

验收：

1. `source_scope` 正确保留
2. PDF、表格、KB 证据进入统一合成
3. 输出不是分段拼接壳子

## Test Plan

必须新增或补齐以下测试。

### 1. Prompt / Routing-Level Tests

1. compare 显式关键词命中时走 compare-aware prompt 分支
2. compare 隐式意图命中时也走 compare-aware prompt 分支
3. 多篇 PDF 已选中但问题只问其中一篇时，不误入 compare 分支
4. 非 compare 问题继续走普通 PDF QA prompt 分支
5. `include_kb=true` 时沿用与 `fastQA` 一致的 KB 使用边界
6. prompt builder 复用方式必须通过 parity snapshot / contract tests 证明，不接受只看人工代码评审

### 2. Multi-PDF Content Tests

1. 多篇 PDF 内容组织后保留清晰分隔头
2. 多篇 PDF 被送入模型前不会丢掉后置文献
3. 均衡截断后，每篇文献都保留最小可用上下文
4. compare 场景下，最终送入模型的文本中必须能断言每篇文献都保留了文件标识和至少一段比较相关证据
5. 预算不足时返回显式失败，而不是静默截掉后置文献继续生成

### 3. Streaming / Steps Contract Tests

1. 文件 QA 流式响应包含 steps 和正文 chunk
2. compare 场景 steps 中包含多文献比较阶段
3. 非流式聚合结果也带完整 steps
4. 流式与非流式的最终 steps 状态一致
5. compare / hybrid 失败时先发失败 step，再发 terminal 事件
6. compare 失败后不允许留下一段看似成功完成的正文

### 4. Fallback Tests

1. compare 场景失败时不会退化成单篇总结
2. 文件不可读时返回可解释错误
3. 仅一篇提取成功时返回“无法完成完整比较”的说明
4. hybrid 场景下不会用 KB 静默补出 PDF 中不存在的结论
5. PDF / 表格 与 KB 冲突时会显式说明冲突来源

### 5. End-to-End Acceptance Tests

1. `mode=patent` 单篇 PDF 总结
2. `mode=patent` 两篇 PDF 比较
3. `mode=patent` PDF + KB 混合问答
4. `mode=patent` PDF + Table + KB 混合问答

其中 `mode=patent` 两篇 PDF 比较必须使用 adversarial fixture，并校验：

1. 答案中至少有一条事实来自文献 A 独有内容
2. 答案中至少有一条事实来自文献 B 独有内容
3. 最终模型输入或中间调试产物可证明两篇文献都进入了 compare 生成阶段

所有与测试相关的命令都应在提权环境运行，不在沙箱环境声称通过。

---

## Rollout Guidance

推荐按以下顺序实施：

1. 先收敛 prompt 与多文献输入组织
2. 再收敛 multi-doc truncation
3. 再把 compare-aware 检测与 prompt 分支收紧到可测试合同
4. 再收敛 streaming / steps / fallback
5. 最后完成混合 QA 统一证据合成
6. 用 adversarial fixtures 和验收样例逐项验证，不允许跳过“对比两篇文献”这个核心场景

---

## Success Criteria

只有同时满足以下条件，本次修复才算完成：

1. `patent` 文件 QA 的 prompt 规则实质复用了 `fastQA` 核心规则
2. 多 PDF 比较问题稳定看到两篇及以上文献参与回答
3. 测试能证明每篇参与比较的文献都贡献了至少一条可区分事实，而不是只出现文件名
4. 混合 QA 不再输出拼接式壳子答案
5. 混合 QA 对 PDF / 表格 / KB 的优先级与冲突规则可被测试验证
6. 流式输出和 steps 行为与 `fastQA` 对齐
7. compare 失败时不会伪装成单篇摘要成功
8. 通过 parity snapshot / contract tests 锁定 `patent` 与 `fastQA` 的共享合同
9. 自动化测试覆盖上述核心场景

只做到“请求落到了 patent 服务”或“返回了一段看起来像答案的文本”，都不算完成。
