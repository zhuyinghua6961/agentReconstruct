# fastQA 近期改动摘要（问答链路）

面向汇报：仅概括**知识库问答／生成式 RAG**相关行为演进，不涉及具体代码与文件清单。  
存储、上传管道等基础设施若未直接改变问答策略，此处不展开。

---

## 1. 检索与排序（Stage2）

- **双路证据对齐**：在向 PDF / Markdown 段落做后续的「再排序」池里，前置合并 **Stage2 向量检索命中的短文段**，减轻「仅从 PDF 前部起手」与用户问题脱节的问题。
- **DOI 归一**：在检索结果与路由键之间对齐 **同一种 DOI 写法**（如斜线与下划线等形态），避免因标识不一致导致的门控落空或重复篮子。
- **焦点词与 DOI 重排**：在纯向量序之外引入**词面与用户问题主轴是否一致**的规则，弱化「只靠辅助检索主张拼进来」导致的题材漂移；同时保留**放松策略**，避免门槛过严时无文献可用。
- **高压实 / 振实等术语细分**：对用户问题里可能混用的「压实（极片）」与「振实（粉体）」等表述做**词面分组**，在焦点扩展时减少把不相关轴线的文献顶到最前。

---

## 2. 查询构造与向量化（进入 embedding 之前）

- **检索句与关键词收口**：对进入向量的检索字符串做更确定性的拼装与截断策略，并与环境项中的上限、注入位配合，减少噪声词挤占 embedding 带宽。
- **嵌入客户端**：对调用方式做小幅稳健化（与超时、批处理或配置约定相关，以实际部署配置为准）。

---

## 3. Stage1 规划与提示

- **规划阶段加强**：提问解析与规划侧补全校验，使后续检索主张、对比类意图等与 Stage2 更一致。

---

## 4. 合成与后处理（Stage4 及后置）

- **流式合成与两阶段路径**：对齐流式输出时序与双阶段合成相关默认策略，降低首包/收尾与引用块错位风险。
- **知识库会话上下文**：将 **KB 问答所需的对话侧上下文**显式传入最终合成阶段，使多轮追问不丢约束。
- **DOI／引用边界与 Markdown**：在句级切分、后处理中提取或修复引用时，**尽量不破坏 Markdown 结构**；收紧 DOI 与参考文献链接的边界，减少误伤正文或列表。
- **引用与 top-k 覆盖校验**：对「检索 top 与最终答案引用」做一致性审计时，使用**规范化后的 DOI** 比较，避免仅因书写形态不同而误报未引用。

---

## 5. 重排、缓存与性能（仍属问答体验）

- **可选本地 rerank 提供方**：在证据重排环节支持走本地实现，降低对外部服务依赖与抖动。
- **Stage2 等阶段缓存签名**：缓存键纳入与检索配置相关的要素，减少错误命中旧检索结果。
- **HTTP 连接与 LLM 传输池**：在多次规划/检索/合成调用之间复用连接，降低高并发下的尾部延迟。

---

## 6. 图谱与其它增强（若业务已开启 Graph KB）

- **Graph KB 升级与路由**：问题可走图谱增强的检索或回答路径（是否启用以实际环境开关为准）。

---

## 7. 可放在汇报里的一句话

**近期 fastQA 问答侧重点在：检索结果更贴用户主轴（焦点词门控与 DOI 归一）、Stage2 密集命中并入 PDF/MD 再排序池、向量化检索句更可控、合成与 DOI／Markdown 后处理更稳，并配套 rerank／缓存／连接层面的体验与性能收口。**
