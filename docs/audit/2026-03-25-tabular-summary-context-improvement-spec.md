# 表格 Summary 模式上下文增强建议

日期：2026-03-25
范围：`fastQA/app/modules/qa_tabular/*`
目的：提出一个不改变“全表执行 + 压缩传给 LLM”总体方向、但显著增强 summary 质量的改造思路

---

## 1. 设计目标

目标不是简单地把整张表原始数据全部塞给 LLM。

目标应当是：

1. 保留“执行层全表计算”的正确模式
2. 避免大表直接原样入 prompt 导致 token 膨胀
3. 让 LLM 真正拿到“足以代表整表”的结构化信息
4. 让生成结果能明显体现“这是基于全表总结”，而不是基于前 5 行样例描述

---

## 2. 为什么不建议直接把全表原始数据传给 LLM

### 2.1 对小表是可行的

如果数据量很小，例如：
- 20 行以内
- 列数有限
- 用户明确要求逐行总结

那么全表原始传递是可以接受的。

### 2.2 对中大表不合理

对于上百行甚至上千行的表，直接传全表原始记录会带来：

1. prompt 急剧膨胀
2. 首 token 时间变慢
3. 推理成本上升
4. 模型更难抓住真正重要的模式
5. 流式输出体验下降

因此更合理的方案是：
- 传“结构化的全表摘要”
- 而不是“全表原始逐行文本”

---

## 3. 推荐的 summary 上下文结构

建议把 `summary` 模式传给 LLM 的上下文从现在的：
- 行数
- 列数
- 列名
- 前 5 行样例

增强为以下 6 层。

### 3.1 表级概览

必须包含：

1. 总行数
2. 总列数
3. 工作表名称
4. 关键列数量
5. 数值列数量
6. 类别列数量
7. 时间列数量
8. 缺失值总体比例

作用：
- 给模型一个真正的整表规模感
- 让它知道表格结构复杂度

### 3.2 列级画像

对每一列或关键列，给出简要画像：

1. 列名
2. 类型
3. 是否数值列
4. 是否类别列
5. 缺失率
6. 去重值数量
7. 样例值

作用：
- 让模型理解这张表里每列扮演什么角色
- 避免只从前几行猜测字段含义

### 3.3 数值列统计摘要

对重要数值列，建议至少提供：

1. min
2. max
3. mean
4. median
5. std 或 IQR
6. 非空数
7. 异常值数量（如有）

作用：
- 真正支撑“整体趋势”“波动范围”“异常程度”这类总结
- 这是 summary 模式最缺的部分之一

### 3.4 类别列分布摘要

对类别列，建议提供：

1. unique count
2. top 3 或 top 5 频次分布
3. 占比

例如：
- 供应商分布
- 状态分布
- 异常标记分布

作用：
- 让模型可以说出“主要集中在哪些类别”“异常主要出现在哪类对象”

### 3.5 代表性样例，而不是简单前 5 行

当前最大的问题不是“样例不该有”，而是“样例选择太弱”。

建议样例从“前 5 行”改成“代表性样例集”，例如：

1. 第一条记录
2. 中位附近记录
3. 最大值记录
4. 最小值记录
5. 异常记录
6. 不同类别代表记录

作用：
- 样例仍然可控
- 但比机械前 5 行更能代表整表

### 3.6 明确区分“总体统计”和“样例”

建议 prompt 中明确写：

1. 以下统计摘要来自全表计算
2. 以下样例仅用于帮助理解数据结构，不代表全部数据

作用：
- 减少模型误把样例当整体
- 降低“只根据 5 条分析”的表述风险

---

## 4. 推荐的 summary prompt 结构

建议 prompt 至少分成以下区块：

1. 用户问题
2. 表级概览
3. 列级画像
4. 数值列统计摘要
5. 类别列分布摘要
6. 代表性样例
7. 明确回答要求

其中回答要求建议写清：

1. 优先基于全表统计给结论
2. 样例只能作为补充说明
3. 不要把样例误表述为全量结果
4. 若某列统计不足，则显式说明

---

## 5. 建议按问题类型动态决定传什么

不是所有表格问题都应该走同一套上下文。

### 5.1 `summary`

重点传：
- 表级概览
- 列画像
- 数值统计
- 类别分布
- 代表性样例

### 5.2 `count_rows`

重点传：
- 过滤条件
- 命中总数
- 若需要则附少量样例

### 5.3 `filter_rows`

重点传：
- 过滤条件
- 命中总数
- 样例
- 截断说明

### 5.4 `groupby`

重点传：
- 分组列
- 聚合结果
- top-k 或完整聚合结果

### 5.5 `topk`

重点传：
- 排序列
- 排序方向
- top-k 全量结果

### 5.6 `trend`

重点传：
- 横轴
- 指标列
- 起点/终点/变化量/方向
- 必要时附短序列

因此真正需要增强的重点其实是 `summary`，而不是整个 tabular QA 全链路。

---

## 6. 推荐的最小改造方向

如果只做最小增强，建议按这个顺序：

### 第一层：保留当前执行框架不动

不改：
- planner
- executor 主体操作定义
- 路由
- SSE 流程

只增强 `summary` 的结果摘要结构。

### 第二层：给 `summary_stats` 补结构化全表统计

建议优先增加：

1. 数值列摘要
2. 类别列 top-k 分布
3. 缺失值概览
4. 列画像摘要

### 第三层：把 `result_rows` 从“前 5 行”换成“代表性样例”

即使总数仍控制在 5 条，也比现在效果更好。

### 第四层：在 prompt 中明确声明“样例 != 全量”

这是低成本但收益很高的一步。

---

## 7. 不建议的改法

### 7.1 不建议直接把整张表转成大段文本喂给 LLM

问题：
- 慢
- 贵
- 不稳定
- 大表几乎不可控

### 7.2 不建议只把样例条数从 5 提到 20 就算解决

这只是把问题延后，不是解决。

如果没有统计摘要，20 条样例也不等于“真正基于全表”。

### 7.3 不建议让 LLM 自己从原始表格文本中推导所有统计

因为后端本来就已经有 dataframe 执行能力，统计应该在执行层得到，而不是浪费 LLM 推导。

---

## 8. 推荐的验收标准

如果未来要改这块，建议用下面标准验收：

1. 用户问“总结这个表格”时，答案能明确体现全表规模
2. 答案能提到关键数值列的范围/均值/异常情况
3. 答案能提到关键类别分布
4. 不再出现明显像“只根据几条样例”得出的错觉
5. 大表场景下首 token 和总耗时仍可接受
6. prompt 大小增长受控

---

## 9. 最终建议

这块真正合理的方向不是：
- “把全表原始数据直接传给 LLM”

而是：
- “把全表执行结果转成结构化统计摘要 + 代表性样例，再传给 LLM”

这样既能保持：
- 全表执行
- token 可控
- 流式体验可控

又能解决当前最大的质量问题：
- 模型只拿到前 5 行样例，导致总结看起来不像基于整表


---

## 10. 本轮已实施的最小增强

本轮实现只覆盖 `summary` 模式，未改变其它表格操作分支。

已实施内容：

1. `executor` 在 `summary_stats` 中新增：
- `column_profiles`
- `numeric_summaries`
- `categorical_summaries`

2. `renderer` 在 `summary` prompt 中新增：
- `全表统计摘要`
- `列画像摘要`
- `数值列摘要`
- `类别列分布摘要`
- `下方仅展示少量代表性样例，不代表全部数据`

3. 保持原有约束：
- 仍然是全表执行
- 仍然控制样例条数，避免 prompt 无限制膨胀
- 未修改 `count_rows / filter_rows / groupby / topk / trend` 等其它模式
