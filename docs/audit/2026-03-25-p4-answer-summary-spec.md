# 2026-03-25 P4-1 答案强制总结能力探索 Spec

## 1. 背景

当前 `fastQA` 与 `highThinkingQA` 的答案是否带“总结段”，更多依赖模型自由发挥。

这会带来两个问题：
- 同类问题在不同模式下输出风格不一致
- 用户想快速抓结论时，无法稳定拿到一个清晰的压缩结论块

用户希望把“总结”从偶然出现，提升为可预测能力。

---

## 2. 目标

本项探索的目标不是立刻强行改 prompt，而是先明确：
- 两条链路是否都应该稳定输出总结块
- 总结块应该在什么阶段生成
- 如何不破坏流式体验、引用对齐和 markdown 渲染

最终希望形成一个可以直接进入实现的收敛方案。

---

## 3. 非目标

本轮不直接承诺这些事情：
- 不重写整个答案模板系统
- 不把总结能力和长期 memory/summary 体系混在一起
- 不要求总结块一开始就支持复杂的前端折叠交互
- 不要求总结块第一版就完全多语言/多样式可配置

---

## 4. 当前现状

### 4.1 `fastQA`
- 主体答案以当前生成链路为主
- 最终 `done` 阶段已有结构化收尾，但没有稳定的“总结块契约”
- 如果直接在 `done.final_answer` 后拼接内容，要小心破坏当前 markdown 渲染与流式一致性

### 4.2 `highThinkingQA`
- 更容易自然生成“结论性段落”，但仍然不稳定
- 它还有草稿、引用检查、最终收尾等阶段，如果总结块插入位置不对，可能进一步放大收尾耗时问题

### 4.3 共同限制
- 不能明显拖慢首 token
- 不能让引用链接和正文脱节
- 不能让前端在 streaming 完成后再次出现格式退化

---

## 5. 方案对比

### 方案 A：纯 prompt 约束
在主 prompt 中明确要求“答案最后必须输出总结”。

优点：
- 实现最轻
- 基本不需要改 SSE 契约
- 两个后端都容易试验

缺点：
- 不稳定，模型可能漏掉或风格漂移
- 很难保证总结块格式完全一致
- 可能压缩主体答案 token budget

适用性判断：
- 适合作为快速 PoC
- 不适合作为最终强约束方案

### 方案 B：流式末尾专门生成总结块
正文流式输出后，再进入一个“总结生成”小阶段，把总结块继续流出来。

优点：
- 不会拖慢正文首 token
- 总结块位置稳定
- 前端能自然复用现有流式渲染能力

缺点：
- 需要定义总结块的 SSE / markdown 契约
- 可能增加收尾时长
- 要避免“正文刚结束，用户以为结束了，结果又冒出一段”的体验问题

适用性判断：
- 是比较平衡的主推方案
- 适合在主链稳定后落地

### 方案 C：done 后结构化补摘要
主体答案流完后，服务端单独产出 `summary_block`，前端在 done 后拼到答案卡片中。

优点：
- 主体流式完全不受影响
- 总结块结构最稳定
- 前端可单独控制样式和折叠

缺点：
- 前端和后端都要改契约
- 如果处理不好，会出现“流式结束后 UI 跳变”
- 需要处理 summary 与 citation 的关系

适用性判断：
- 长期最干净
- 但前后端联动改造成本最高

---

## 6. 推荐方案

推荐采用“两阶段推进”：

### 第一阶段：先做方案 A 的受控实验
目标：
- 快速验证用户是否真的需要稳定总结块
- 观察对答案质量和冗余度的影响

要求：
- 只在内部开关下启用
- 日志记录总结块是否出现、长度、是否带引用
- 不承诺最终产品形态

### 第二阶段：如果实验收益明确，收敛到方案 B
原因：
- 比纯 prompt 更稳定
- 比 done 后拼接更不容易引入前端跳变
- 保持“全程仍然是流式生成”的产品体验

不推荐直接上方案 C 作为第一刀。

---

## 6.1 当前收敛决策

结合 [当前输出契约审计](/home/cqy/worktrees/highThinking/docs/audit/2026-03-25-answer-summary-current-contract.md)，本轮先收敛以下决策：

- 第一阶段仍走方案 A，但“总结块”必须进入最终权威正文，而不是只出现在中间流式内容
- 不新增 `summary_block` 之类的前端结构化字段
- 前端继续只消费 `done.final_answer`
- `gateway` 不作为第一阶段改造点

原因很直接：
- 前端完成态会用 `done.final_answer` 覆盖流式正文
- `gateway` 持久化也以 `done.final_answer` 作为最终助手消息
- 如果实验产物没有进入最终权威答案，用户完成态和持久化都会失真

---

## 6.2 第一阶段协议

### 协议 1：总结块归属

第一阶段把总结块定义为“最终 markdown 正文的一部分”，不是单独字段。

这意味着：
- `fastQA`：总结块必须进入 `final_result.final_answer`
- `highThinkingQA`：总结块必须进入 `state.final_answer`
- `done.final_answer` 必须携带总结块完整正文

### 协议 2：总结块位置

统一放在答案末尾，不放开头，不做中段插入。

原因：
- 不打断用户先读主体结论和论证
- 不影响当前 `content -> done.final_answer` 的完成态覆盖逻辑
- 最容易和现有 markdown 渲染保持一致

### 协议 3：标题与格式

第一阶段固定成 markdown 二级标题：

`## 总结`

正文二选一：
- 2 到 4 句短段落
- 或 3 到 5 个 bullet

但不能混成第二个长答案。

### 协议 4：引用策略

第一阶段不强制总结块带引用。

约束是：
- 如果主体答案已经有引用，优先复用主体中已经出现的引用
- 不为了总结块新增一组全新的 DOI
- 不允许总结块为了“看起来更完整”而引入正文未使用的新证据

### 协议 5：短答案豁免

当主体答案本身已经非常短时，不强制补总结块。

建议豁免条件：
- 主体正文已经是 3 句以内直接结论
- 或答案本身已经以“总结/结论”结构输出

原因：
- 否则很容易把简洁答案做成重复答案

### 协议 6：失败降级

总结块生成失败时：
- 主体答案必须正常返回
- 不得阻断 `done`
- 只允许记日志，不允许把整条问答打成失败

### 协议 7：日志埋点

第一阶段至少记录：
- `summary_enabled`
- `summary_generated`
- `summary_format`
- `summary_length`
- `summary_has_citation`
- `summary_generation_ms`
- `summary_skipped_reason`

---

## 7. 关键设计问题

### 7.1 总结块是否必须带引用
建议：
- 第一版允许“不强制引用”
- 但如果主体答案已有引用，优先复用已有文献而不是引入新引用

### 7.2 总结块长度
建议：
- 控制为 2 到 4 句
- 或者 3 到 5 个 bullet，不要无限制展开

### 7.3 总结块位置
建议：
- 放在主体答案末尾
- 标题统一，如 `总结` / `结论摘要`
- 不建议放在答案开头，避免打断阅读顺序

### 7.4 两条链路是否完全统一
建议：
- 统一“有无总结块、位置、标题、长度约束”
- 不强求两条链路的语气完全一样

### 7.5 两条链路的插入边界
建议：
- `fastQA` 在真正形成 `final_answer` 的收尾边界插入，而不是只追加到流式 chunk
- `highThinkingQA` 在 checker/reviser 结束后的最终答案边界插入，而不是只追加到 draft

这是第一阶段最重要的实现边界。

## 9. 验收标准

满足以下条件，才认为可以进入开发：
- 已明确总结块采用哪种生成时机
- 已明确是否带引用及其约束
- 已明确前端最终渲染不会在结束后退化
- 已明确不会明显损害首 token 与正文输出平滑度

---

## 10. 风险与依赖

依赖：
- 当前 `fastQA` / `highThinkingQA` 的流式输出契约稳定
- 当前前端 markdown 渲染收尾行为稳定

主要风险：
- 总结块把“简洁”变成“重复”
- 服务端为了稳定总结块，引入额外收尾延迟
- 引用关系在总结块里再次漂移
