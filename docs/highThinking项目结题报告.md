## 二、项目概述

highThinking 项目面向科研问答、文献问答、文件问答、图谱问答和专利问答等复杂场景，构建了一套可拆分、可部署、可扩展的多模式智能问答平台。项目从原先较强耦合的业务系统中，逐步抽象出统一前端、统一网关、公共能力后端和多个模式化 QA 后端，使系统能够在同一个用户入口下支持快速问答、深度思考问答、文件/混合问答和专利问答。

系统当前采用“前端 -> Gateway -> 公共服务/QA 服务”的架构。前端只面向网关发起请求；网关负责鉴权透传、公共接口代理、模式路由、文件上下文解析、SSE 流式转发和任务调度；公共服务负责认证、会话、文件、文档、配额、后台管理等公共能力；fastQA、highThinkingQA 和 patent 分别承接不同问答模式的执行逻辑。

项目已经形成较完整的工程交付物：前后端源码、服务启动脚本、统一配置目录、部署配置、Docker 离线交付方案、数据包 seed 方案、自动化测试以及大量架构/迁移/验收文档。整体上，项目已具备阶段性结题和继续迭代交付基础。

## 三、项目背景与建设必要性

科研和专利资料问答系统面临三个典型问题。

第一，业务能力和问答能力容易耦合。用户认证、会话持久化、文件上传、PDF 查看、配额管理、后台管理等能力与问答链路混在同一个后端中，会导致后续扩展新问答模式时重复建设。

第二，不同问答模式的执行链路差异明显。快速问答强调响应速度和文件/表格处理能力；深度问答强调多阶段推理、子问题分解、检索与修订；专利问答还需要结构化图谱、原文查看和专利字段理解。若全部放在一个服务中，维护成本和故障影响面都会变大。

第三，实际部署需要稳定的工程闭环。系统不仅要能回答问题，还要解决流式输出、失败/取消持久化、文件状态、配额扣减、配置隔离、离线部署、数据导入和服务健康检查等工程问题。

因此，本项目的建设重点不是单一模型调用，而是把智能问答能力工程化为可运行、可治理、可部署的多服务系统。

## 四、项目目标

### 4.1 总体目标

建设一个统一入口、多模式路由、公共能力集中、问答执行服务独立的智能问答平台，使用户能够在同一个前端中完成登录、会话管理、资料上传、文献查看、普通问答、文件问答、深度问答和专利问答等操作。

### 4.2 具体目标

1. 建设统一前端，提供问答、会话、上传、PDF 阅读、引用查看、用户中心、管理员后台和配额管理能力。
2. 建设 Gateway/BFF，统一承接浏览器请求，负责公共接口代理、多模式 QA 路由、文件上下文解析和 SSE 流式透传。
3. 抽取公共服务，集中管理认证、会话、文件、文档、配额、人员、部门、系统状态等公共能力。
4. 拆分问答后端，使 fastQA、highThinkingQA、patent 分别承担清晰的模式化问答职责。
5. 建立统一运行资源目录，规范配置、状态、运行日志、共享资产和部署数据。
6. 提供可测试、可启动、可打包的工程交付方式，支撑本地联调与离线部署。

## 六、总体架构

项目采用多服务分层架构：

```text
用户浏览器
  -> frontend-vue
    -> gateway
      -> public-service
      -> fastQA
      -> highThinkingQA
      -> patent
      -> MySQL / Redis / MinIO / Neo4j / 向量数据库
```

### 6.1 前端层

前端位于 `frontend-vue/`，是当前正式用户入口。它通过 `/api/*` 访问网关，默认开发端口为 `5173`。主要能力包括：

- 会话列表与多会话切换；
- 流式问答展示；
- PDF / Excel 上传；
- 文献引用、DOI 面板、PDF 打开与参考文献预览；
- 本地会话状态持久化；
- 登录、注册、找回密码、个人中心；
- 管理员用户管理、部门/人员管理、配额管理、模型状态查看。

### 6.2 网关层

网关位于 `gateway/`，默认端口为 `8101`。它是浏览器面对的唯一后端入口，主要负责：

- 代理公共能力接口到 `public-service`；
- 提供 `POST /api/{mode}/ask` 与 `POST /api/{mode}/ask_stream`；
- 解析用户请求中的文件上下文；
- 在 `fast`、`thinking`、`patent` 模式之间做执行后端选择；
- 对文件问答和混合问答进行路由收口；
- 维持 SSE 流式传输不被缓冲到完成后再返回；
- 提供任务、取消、队列、准入控制和健康检查相关能力。

### 6.3 公共服务层

公共服务位于 `public-service/backend/`，默认端口为 `8102`。它负责所有不应由 QA 后端重复实现的公共能力，主要模块包括：

- `system`：健康检查、知识库状态、缓存刷新、后台状态、模型状态；
- `auth`：登录、注册、用户信息、密码、安全问题、人员绑定；
- `admin_users`：管理员用户管理、批量导入、状态和类型管理；
- `departments` 与 `personnel`：组织部门和人员信息管理；
- `quota`：配额配置、用户配额、预检查和扣减确认；
- `conversation`：会话、消息、文件元数据、JSON 主文档、缓存和内部 authority 接口；
- `documents`：PDF 查看、原文查看、摘要、翻译、参考文献预览；
- `uploads`：PDF/Excel 上传、对象存储镜像、会话文件登记。

### 6.4 QA 执行层

QA 执行层由多个后端组成：

- `fastQA/`：承接 fast 模式，重点负责快速知识库问答、文件问答、表格问答、混合问答和部分图谱增强链路。
- `highThinkingQA/`：承接 thinking 模式，保留多阶段高质量推理问答能力，包括问题分解、子问题回答、检索、综合、Checker/Reviser 修订等。
- `patent/`：承接 patent 模式，围绕专利资料、专利原文、专利表格、专利图谱和专利 RAG 进行建设。

### 6.5 资源与部署层

`resource/` 作为共享资源根目录，划分为：

- `resource/config/`：共享配置与各服务配置模板；
- `resource/assets/`：只读共享资产；
- `resource/state/`：可持久化状态；
- `resource/runtime/`：pid、日志、临时运行文件；
- `resource/mounts/`：外部挂载约定；
- `resource/scripts/`：跨服务运行脚本。

`deploy/` 提供 Docker 离线部署能力，覆盖业务服务、前端 nginx、MySQL、Redis、MinIO、两个 Neo4j 图谱库和数据 seed 任务。

## 七、主要功能实现

### 7.1 统一用户入口

项目以 `frontend-vue/` 为正式前端入口，实现了用户登录、注册、找回密码、问答主页、个人中心、管理员后台和配额管理等页面。前端通过网关统一访问后端，避免直接耦合多个服务地址。

### 7.2 多模式问答

系统支持 `fast`、`thinking`、`patent` 三类模式化问答入口。网关对外提供统一的问答 API，内部再按模式与问题类型转发到不同后端。

- fast 模式：强调响应速度、文件参与问答、表格处理、混合问答和图谱增强。
- thinking 模式：强调高质量推理、多阶段生成、引用校验和答案修订。
- patent 模式：强调专利原文、专利结构化字段、专利图谱和专利技术语义。

### 7.3 文件 QA 与混合 QA

项目明确了文件参与问答的范围和判定规则，覆盖：

- `pdf_qa`：仅基于 PDF 内容回答；
- `tabular_qa`：仅基于表格内容回答；
- `hybrid_qa`：联合 PDF、表格、知识库中至少两类来源回答。

网关负责判断用户问题是否需要文件参与，并将文件问答或混合问答收口到合适执行后端，减少后端重复解析文件意图。

### 7.4 知识库与 RAG 链路

fastQA 与 highThinkingQA 都围绕知识库问答建立了阶段化链路。fastQA 侧重点在检索、重排、缓存、文件链路和响应速度；highThinkingQA 侧重点在深度推理、多阶段分解与答案修订。

问答链路中包含：

- 用户问题解析；
- 查询规划；
- 向量检索；
- 文档或图谱证据组织；
- LLM 综合生成；
- 引用、DOI、Markdown 与最终答案后处理；
- SSE 流式输出；
- 失败、取消和完成状态持久化。

### 7.5 图谱增强问答

项目引入 Neo4j 图数据库支持图谱问答。fastQA 与 patent 均具有图谱增强路径。图谱链路不直接替代 RAG，而是按问题类型选择直接图谱回答、图谱约束 RAG 或降级到普通向量 RAG。

专利图谱方向已形成专利实体、工艺步骤、材料、实验表、技术问题、技术方案、应用场景、发明点、性能事实、保护范围、IPC、组织和人员等结构化查询目标。

### 7.6 会话与持久化

项目将公共会话能力逐步收口到 `public-service`，形成统一 conversation authority。会话能力覆盖：

- 会话创建、列表、详情、删除和标题更新；
- 用户消息和 assistant 消息写入；
- 会话文件元数据；
- JSON 主文档；
- Redis 缓存；
- outbox 和远端同步退化机制；
- 失败/取消终态 assistant 消息持久化。

### 7.7 配额管理

项目统一了用户可理解的配额桶：

- `ask_query`：普通问答；
- `file_qa`：文件问答；
- `file_view`：查看原文；
- `doc_assist`：文档辅助。

公共服务作为配额 authority，网关和执行服务按预检查、执行、成功后确认扣减的流程协作，避免失败请求错误扣费。

### 7.8 流式输出与取消

系统使用 SSE 承载问答流式输出。网关负责无缓冲透传，后端负责按阶段输出元数据、步骤、内容、错误和完成事件。项目还加入任务、取消、终态持久化和前端恢复逻辑，使刷新后仍能看到失败或取消的历史消息。

### 7.9 模型接入与思考控制

项目近期形成了统一 LLM 配置与 thinking 控制设计。目标是通过 `LLM_BASE_URL`、`LLM_API_KEY`、`LLM_MODEL`、`LLM_IS_THINKING_MODEL`、`LLM_THINKING_ENABLED` 等少量配置控制 OpenAI-compatible 模型调用。对规划、检索、翻译、JSON 等控制面调用默认禁用 thinking，只允许最终答案阶段按配置启用 thinking，避免 reasoning 内容泄露到用户可见输出。

### 7.10 离线部署

`deploy/` 目录提供面向交付的 Docker 离线部署方案。部署包覆盖：

- gateway；
- public-service；
- fastQA；
- highThinkingQA；
- patent；
- frontend nginx；
- edge nginx；
- MySQL；
- Redis；
- MinIO；
- Neo4j literature；
- Neo4j patent；
- seed tools 与数据包导入任务。

数据包采用 `deploy/data/*.tar.zst` 方式交付，镜像和大数据分离，降低镜像体积并提升离线部署可控性。

## 八、关键技术特点

### 8.1 网关统一路由

系统通过 Gateway 将前端和多个后端解耦。公共接口固定代理到公共服务，QA 接口按模式和文件上下文路由到不同问答服务。该设计降低前端复杂度，也为未来新增模式提供扩展点。

### 8.2 公共能力独立化

认证、会话、上传、文档、配额和后台管理被集中到 `public-service`，避免 fastQA、highThinkingQA、patent 重复维护同一类业务代码。公共能力独立化是项目从单体走向多服务的核心成果之一。

### 8.3 文件上下文标准化

前端发送原始文件上下文，网关转化为后端可执行的标准化请求，包含实际模式、路由、来源范围、文件选择、路由原因和置信度等字段。该机制使后端更专注于执行，而不是重复做文件意图解析。

### 8.4 阶段化 RAG

问答链路被拆为规划、检索、证据组织、综合生成、引用修复和最终输出等阶段，有利于缓存、观测、单元测试和性能优化。

### 8.5 图谱与向量协同

图谱用于结构化事实查询、候选约束和证据增强，向量 RAG 用于语义检索和综合生成。两者不是互斥关系，而是根据问题类型组合使用。

### 8.6 工程可交付性

项目不只包含算法链路，还包含本地启动、健康检查、配置管理、日志、运行目录、Docker 镜像、离线数据 seed、部署前检查等工程能力，具备继续走向现场交付的基础。

## 十二、存在问题与风险

项目当前已达到阶段性交付状态，但仍存在以下待完善点：

1. 多服务端到端联调仍需继续加强。现有服务均有独立测试与接口契约，但 gateway 到 public-service、fastQA、highThinkingQA、patent 的真实部署路径仍需要更多 live smoke 验证。
2. highThinkingQA 仍保留部分公共能力路由，后续应进一步收敛为 thinking QA 执行服务，避免与 public-service 职责重叠。
3. fastQA 与 patent 的部分问答质量、图谱路径和资源路径仍依赖具体数据包、向量库和图谱库，需要在目标部署环境中验证数据完整性。
4. 模型服务稳定性是系统运行关键依赖。若 OpenAI-compatible 网关或模型后端返回 502、超时或协议不兼容，问答能力会受影响，需要配套模型服务健康检查和日志追踪。
5. 离线部署已经具备方案，但现场部署仍需要证书、端口、模型 endpoint、密钥、数据包版本和硬件资源的最终确认。
6. 前端长会话、流式渲染、大文件上传、并发问答、取消恢复等体验场景仍建议持续做人工验收和压力测试。

## 十三、后续工作建议

1. 补齐全链路验收脚本：覆盖登录、创建会话、上传文件、文件列表、普通问答、文件问答、thinking 问答、patent 问答、取消和刷新恢复。
2. 收敛公共能力 ownership：以 public-service 为唯一权威，逐步清理 highThinkingQA 等旧路由中的公共能力实现。
3. 完善部署环境观测：增加模型服务、Redis、MinIO、MySQL、Neo4j、向量库和各后端的统一健康看板。
4. 固化数据包验收：为 MinIO 原文、向量库、Neo4j dump、专利原文和公共服务 reference 数据建立版本校验清单。
5. 加强性能与稳定性测试：重点验证高并发问答、长会话渲染、SSE 不中断、模型 502/超时降级和取消任务清理。
6. 梳理正式验收材料：补充项目人员、工期、经费、现场部署截图、测试报告、用户试用意见和最终验收签字页。

## 十四、结题结论

highThinking 多模式智能问答平台已经完成从单体业务系统向多服务智能问答平台的阶段性重构，形成了统一前端、统一网关、公共能力服务和多个模式化 QA 后端的工程架构。项目不仅实现了问答能力本身，还围绕会话、文件、配额、流式输出、图谱、部署和测试建立了较完整的工程支撑体系。

从当前仓库可见成果判断，项目已具备阶段性结题条件：核心架构明确，主要服务可独立运行，公共能力已完成大部分拆分，问答模式已形成清晰边界，部署和测试资产齐备。若以正式生产验收为目标，建议在结题后继续围绕全链路联调、现场部署验证、模型服务稳定性和旧职责清理开展收口工作。

综上，本项目达到阶段性建设目标，可作为后续产品化部署、行业场景适配和问答质量优化的基础版本。
